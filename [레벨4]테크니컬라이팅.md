# **📑 안드로이드 XML에서 Jetpack Compose로 점진적 마이그레이션 가이드**

## **0. 서론: Compose 도입 배경과 전략**

우아한테크코스 7기에서는 우선 XML 기반의 안드로이드 화면 작성을 공부했습니다.

이 과정에서 뷰 바인딩(View Binding), 데이터 바인딩(Data Binding) 등의 기술을 활용하여 UI를 제어하는 경험을 쌓았습니다.

저 또한 XML 기반 UI 학습을 통해 화면 구성과 상태 관리를 익혔습니다. 이후 **레벨 4에서 Jetpack Compose를 학습**하게 되었고, 미션 과제로 기존 XML 프로젝트의 일부 화면을 Compose로 전환하는 경험을 하게 되었습니다.

이 과정에서 **작은 단위부터 Compose를 도입하고 점진적으로 화면을 전환**하는 경험을 통해, 점진적 마이그레이션 전략의 효용을 체감할 수 있었습니다.

따라서 본 글의 주제는 **XML 기반 안드로이드 프로젝트를 Compose로 점진적 전환하는 방법**으로 설정하게 되었습니다.

### **Jetpack Compose란?**

Jetpack Compose는 2019년 Google I/O에서 처음 공개된 이후, 안드로이드 개발의 UI 패러다임을 **선언형(UI declarative)**으로 전환한 프레임워크입니다.

XML 기반의 전통적인 UI는 View 트리를 직접 조작하는 명령형 방식으로, 화면 복잡도가 증가함에 따라 **상태 불일치와 UI 갱신의 어려움**이 발생했습니다.

반면 Compose는 **상태(State)를 기반으로 UI를 재구성(Recompose)**하는 선언형 접근 방식을 통해, **더 적은 코드로 일관된 UI를 표현하고 유지보수성을 향상**시킬 수 있습니다.

```kotlin
// 기존 XML 명령형 방식
val textView: TextView = findViewById(R.id.textView)
textView.text = "안녕하세요"

// Compose 선언형 방식
@Composable
fun Greeting(name: String) {
    Text(text = "안녕하세요, $name")
}
```

### **Compose 마이그레이션 전략**

본 글에서 제안하는 Compose 점진적 마이그레이션 전략은 다음과 같습니다.

1. **기존 화면에서 재사용 가능한 컴포넌트부터 Compose로 전환**
    - 화면 내 버튼, 카드, 리스트 아이템 등 **작은 UI 단위**를 Compose로 구현
    - **ComposeView**를 활용하여 기존 XML 레이아웃에 Composable 함수를 삽입
    - 이를 통해 기존 화면 기능을 유지하면서 Compose를 테스트 가능
2. **AndroidView를 활용한 기존 View 재사용**
    - 기존 프로젝트에서 사용 중인 `WebView`, `MapView` 등의 뷰를 Compose 화면 내에서 재사용
    - Compose와 기존 View의 **상호 운용성 확보**를 통해, 기존 기능 안정성 유지
3. **새로운 화면은 처음부터 Compose로 작성**
    - 신규 Activity/Fragment는 전체 화면을 Compose로 구현
    - ViewModel 및 상태 관리 연계, Compose Navigation 적용

위의 전략은 안드로이드 공식문서가 제공하는 [이전 전략](https://developer.android.com/develop/ui/compose/migrate/strategy?hl=ko)을 참고해 작성되었습니다. **작은 단위 → 기존 화면 일부 영역 → 화면 단위 → 전체 화면** 순으로 점진적 전환이 이루어지도록 설계한 것입니다.

이러한 전략을 바탕으로, 이후 장에서는 실제 프로젝트에서 **컴포넌트 단위로 Compose를 도입하는 단계부터, 화면 전체를 Compose로 전환하는 과정까지** 순차적으로 다루겠습니다.

## **1. 개발 환경 준비 및 기존 UI 분석**

우선, Jetpack Compose를 도입하기 위해서는 Gradle 설정에서 Compose 기능을 활성화해야 합니다. 또한, Kotlin 버전과 Compose Compiler 버전의 호환성도 함께 확인해야 합니다.

**(1) build.gradle(:app) 설정**

```kotlin
android {
    buildFeatures {
        compose = true // Compose 활성화
    }
    composeOptions {
        kotlinCompilerExtensionVersion = "version_name" // Compose Compiler 버전 지정
    }
}

dependencies {
    // Compose BOM을 통해 버전 일괄 관리
    implementation(platform("androidx.compose:compose-bom:2025.10.00"))

    // Compose 핵심 라이브러리
    implementation("androidx.compose.ui:ui")
    implementation("androidx.compose.material3:material3")
    implementation("androidx.compose.ui:ui-tooling-preview")
    debugImplementation("androidx.compose.ui:ui-tooling")

    // Activity Compose 지원
    implementation("androidx.activity:activity-compose:1.9.0")
}

```

> 💡 Tip: Compose 라이브러리의 버전 관리는  **Compose BOM(Bill of Materials)**을 활용하면 일관성 있는 종속성 버전 관리가 가능합니다.
>

Gradle 구성이 완료되면, 이제 **어떤 화면부터 Compose로 전환할지**를 판단해야 합니다.

Compose로의 전환은 앞서 말했듯이 한 번에 전체를 바꾸기 보다, 작은 단위 → 화면 일부 → 전체 화면의 점진적 접근이 바람직합니다.

이를 위해 기존 XML 기반 UI를 다음 세 가지 관점에서 분석할 수 있습니다.

| 분석 항목 | 설명 | 예시 |
| --- | --- | --- |
| **UI 복잡도** | 중첩된 뷰, 커스텀 레이아웃 등 복잡한 구조 | ConstraintLayout, RecyclerView 등 |
| **상태 의존성** | LiveData, ViewModel에 의존하는 정도 | 상태 변화가 잦은 화면 |
| **재사용성** | 공통적으로 쓰이는 UI 구성요소 | 버튼, 카드, 리스트 아이템 |

이 분석 과정을 통해, 단순한 구성부터 점차 복잡한 화면으로 확장하는 전략을 세울 수 있습니다.

현재 제가 마이그레이션을 진행 중인 **festabook** 프로젝트의 경우, ‘소식’ 화면에서 공지와 FAQ 아이템이 동일한 UI 구조를 가지고 있었습니다. 이 두 컴포넌트는 **재사용 가능성이 높고, 상태 의존성이 낮은 단순 UI 요소**였기 때문에 Compose 전환의 첫 대상으로 선정하였습니다.

## 2. 컴포넌트 단위 Compose 전환: 작은 UI부터 시작하기

Compose 마이그레이션의 첫 단계는 **재사용성이 높고 독립적인 UI 컴포넌트를 Compose로 전환하는 것**입니다.

이는 전체 화면 구조를 건드리지 않으면서도 Compose의 선언적 UI 방식을 도입할 수 있는 가장 안전한 접근입니다.

### **2.1 전환 대상 선정**

Compose로 옮길 대상을 선택할 때는 다음 세 가지 기준을 활용했습니다.

| 기준 | 설명 | 예시 |
| --- | --- | --- |
| **재사용성** | 여러 화면에서 반복되는 UI | 카드, 버튼, 리스트 아이템 |
| **독립성** | 복잡한 비즈니스 로직에 의존하지 않는 경우 | 단순 정보 표시용 View |
| **상태 단순성** | 단일 클릭이나 확장 상태 등 단순한 상태 | FAQ, 공지사항 아이템 |

이 기준에 따라, 공지사항과 FAQ 화면에서 사용되는 **아이템(NewsItem)**을 Compose로 분리하기로 결정했습니다.

두 구성요소는 동일한 디자인을 공유하고, 다른 화면에서도 재사용될 가능성이 높았습니다.

### **2.2 NewsItem: XML → Compose 변환**

기존 XML 레이아웃과 `ViewHolder`로 작성되던 카드형 아이템을 Compose의 **`Composable` 함수**로 변환했습니다

아래의 `NewsItem`은 제목, 설명, 아이콘, 생성일 등을 받아 하나의 독립적인 UI로 표현합니다.

```kotlin
@Composable
fun NewsItem(
    title: String,
    description: String,
    isExpanded: Boolean,
    onClick: () -> Unit,
) {
    Column(
        modifier = Modifier
            .fillMaxWidth()
            .clickable { onClick() }
            .animateContentSize()
            .padding(16.dp)
    ) {
        Text(text = title, fontSize = 14.sp)
        if (isExpanded) {
            Spacer(Modifier.height(8.dp))
            Text(text = description)
        }
    }
}

```

이처럼 Compose에서는 클릭 이벤트, 애니메이션, 레이아웃 변경을 XML보다 훨씬 간결하게 표현할 수 있습니다.

### **2.3 기존 XML 화면에 Compose 삽입하기**

화면 전체를 Compose로 바꾸지 않고도 **기존 Fragment나 Activity 내에 `ComposeView`를 추가**하여 점진적으로 도입할 수 있습니다.

**(1) XML 레이아웃에 ComposeView 추가**

```xml
<androidx.compose.ui.platform.ComposeView
    android:id="@+id/newsItemCompose"
    android:layout_width="match_parent"
    android:layout_height="wrap_content" />

```

**(2) Fragment/Activity에서 `setContent` 호출**

```kotlin
binding.newsItemCompose.setContent {
    NewsItem(
        title = "공지사항",
        description = "페스타북 서비스가 업데이트되었습니다.",
        isExpanded = viewModel.isExpanded,
        onClick = { viewModel.toggleExpand() }
    )
}
```

이 방식은 기존 화면 로직을 그대로 유지하면서도 Compose 기반의 UI를 안전하게 통합할 수 있다는 점에서 초기 단계에 매우 적합합니다.

### **2.4 화면 단위의 ComposeView 구성하기**

앞서 살펴본 예시에서는 기존 XML 화면 내 일부 영역에서만 `ComposeView`를 사용했습니다.

하지만, Compose 전환이 익숙해지고 화면 내 의존성이 크지 않다면, **화면 전체를 Compose로 구성하는 중간 단계**로 확장할 수 있습니다.

이 단계에서는 **Fragment 자체의 Root View를 ComposeView로 대체해, Screen 단위로 화면을 대체할 수 있습니다.**  즉, 여전히 기존 XML 기반의 Activity 및 네비게이션 구조를 유지하면서도 **특정 화면은 Compose로 완전히 전환**할 수 있습니다.

아래의 예시는 실제 `festabook` 프로젝트의 `FAQFragment`에서 적용한 코드입니다.

```kotlin
class FAQFragment : BaseFragment<FragmentFaqBinding>(R.layout.fragment_faq) {
    private val viewModel: NewsViewModel by viewModels({ requireParentFragment() }) { NewsViewModel.Factory }

    override fun onCreateView(
        inflater: LayoutInflater,
        container: ViewGroup?,
        savedInstanceState: Bundle?,
    ): View =
        ComposeView(requireContext()).apply {
            setViewCompositionStrategy(ViewCompositionStrategy.DisposeOnViewTreeLifecycleDestroyed)
            setContent {
                FAQScreen(uiState = viewModel.faqUiState, onFaqClick = { faqItemUiModel ->
                    (requireParentFragment() as OnNewsClickListener).onFAQClick(faqItemUiModel)
                })
            }
        }

    companion object {
        fun newInstance() = FAQFragment()
    }
}
```

> 💡 `FragmentFaqBinding`을 상속하지만, 실제로는 **XML 레이아웃을 완전히 사용하지 않고** `ComposeView`를 루트로 반환합니다. XML 파일은 단순히 기존 베이스 클래스를 유지하기 위한 형식적 역할만 하고, UI 렌더링은 모두 Compose에서 수행되고 있습니다.
>

## 3. AndroidView로 기존 View 통합하기

이번에는 반대로, **Compose 화면에서 기존 View를 사용하는 방법**에 대해 알아보겠습니다.

Compose로 화면 전환을 진행하다 보면, 기존 XML 기반에서 사용하던 `WebView`, `MapView`, `VideoView` 등의 **커스텀 View나 외부 SDK View를 완전히 대체하기 어려운 경우**가 있습니다.

특히 오랫동안 사용해온 **외부 SDK 기반 View**(예: 지도, 광고, 비디오 플레이어 등)는 Compose 전용 컴포넌트로의 전환이 쉽지 않거나, 아직 라이브러리 지원이 완전하지 않은 경우가 많습니다.

제가 참여 중인 **festabook** 프로젝트에서도 이러한 문제에 직면했습니다.

앱 내에서 축제 지도를 표시하기 위해 **Naver Map SDK**를 사용하고 있는데, Compose 기반의 Naver Map SDK(`naver-map-compose`)가 이미 존재하긴 하지만, 프로젝트는 기존 SDK를 기반으로 구현되어 있었습니다.

새로운 Compose SDK로 완전히 다시 작성하기에는 개발 공수와 리스크가 컸기 때문에, **“우선 Compose로 화면 전환을 진행하되 지도는 기존 View를 유지하자”** 는 결정을 내렸습니다.

이때 사용할 수 있는 방법이 바로 `AndroidView`입니다.

**AndroidView 기본 구조**

`AndroidView`는 기존 View 인스턴스를 Compose 환경 안에 삽입하는 **브리지(Bridge)** 역할을 수행합니다.

기본 구조는 다음과 같습니다.

```kotlin
@Composable
fun MapContainerView() {
    AndroidView(
        factory = { context ->
            MapView(context).apply {
                getMapAsync { naverMap ->
                    naverMap.uiSettings.isZoomControlEnabled = false
                }
            }
        },
        update = { mapView ->
            // 상태 업데이트 시 필요한 로직 (ex. 카메라 이동)
        }
    )
}
```

- **`factory`** : View를 최초로 생성할 때 호출
- **`update`** : Compose 상태가 변경될 때마다 UI를 갱신

즉, 기존 View를 완전히 폐기하지 않고 **Compose 내부에서 재활용**할 수 있습니다.

특히 festabook처럼 외부 SDK를 사용하는 프로젝트에서는, Compose 도입 속도를 늦추지 않으면서 안정성을 확보하는 현실적인 대안으로 유용하게 사용될 수 있습니다.

---

> 💡 Tip:
>
>
> Compose 1.5 이후에는 `AndroidViewBinding`을 통해 **기존 XML 레이아웃 전체를 Compose 내에서 재사용**할 수도 있습니다.
>
> 예를 들어 다음처럼 ViewBinding을 바로 Compose 안에 넣을 수 있습니다.
>
> ```kotlin
> AndroidViewBinding(ItemFestivalMapBinding::inflate) { binding ->
>     binding.mapView.getMapAsync { ... }
> }
> ```

## 4. 신규 화면 전체를 Compose로 구축하기

이전까지는 기존 XML 프로젝트 내에 Compose를 삽입하는 형태였다면, 이제는 새로운 화면을 Compose 전용으로 작성할 수 있습니다.

`ComponentActivity`와 `setContent {}`를 활용하여 전체 화면을 Composable로 구성하고, Navigation Compose를 통해 기존 화면과 연동할 수 있습니다.

이 단계부터는 더 이상 XML 레이아웃이 필요하지 않으며, Compose 생태계를 중심으로 UI, 상태, 네비게이션을 모두 관리하게 됩니다.

## **5. 공통 UI 컴포넌트 라이브러리화 및 테마 통합**

Compose 전환이 일정 수준 진행되면, 각 화면마다 비슷한 UI 패턴이 반복되기 시작합니다.

이 시점부터는 **UI 컴포넌트를 공통화하고, 테마 시스템을 일관성 있게 관리**하는 것이 중요합니다.

### **5.1 공통 UI 컴포넌트 추출**

Compose에서는 Composable 함수가 재사용 가능한 단위이므로, 공통 UI 요소를 **컴포넌트 단위로 추출하여 모듈화**할 수 있습니다.

예를 들어, 여러 화면에서 반복되는 버튼이나 카드 형태를 다음과 같이 정의할 수 있습니다.

```kotlin
@Composable
fun FestaButton(
    text: String,
    onClick: () -> Unit,
    modifier: Modifier = Modifier,
    enabled: Boolean = true
) {
    Button(
        onClick = onClick,
        enabled = enabled,
        modifier = modifier
            .fillMaxWidth()
            .height(48.dp)
            .clip(RoundedCornerShape(12.dp))
    ) {
        Text(
            text = text,
            style = MaterialTheme.typography.labelLarge,
            color = MaterialTheme.colorScheme.onPrimary
        )
    }
}

```

> 💡 Tip:
>
>
> 공통 컴포넌트는 `ui/component` 패키지에 모아 관리하면, 화면 단위 Composable(`Screen`)과 명확히 구분할 수 있습니다.
>
> 예: `ui/component/FestaButton.kt`, `ui/component/FestaCard.kt`
>

### **5.2 MaterialTheme을 통한 테마 일원화**

Compose의 핵심 장점 중 하나는 **`MaterialTheme`를 통한 전체 UI의 일관된 스타일 관리**입니다.

기존 XML에서는 색상, 폰트, 스타일이 분산되어 있었다면, Compose에서는 하나의 진입점에서 모두 제어할 수 있습니다.

```kotlin
@Composable
fun FestaBookTheme(content: @Composable () -> Unit) {
    val colorScheme = lightColorScheme(
        primary = Color(0xFF0061A4),
        secondary = Color(0xFF5E92F3),
        background = Color.White,
        onPrimary = Color.White,
        onBackground = Color.Black
    )

    val typography = Typography(
        titleLarge = TextStyle(
            fontFamily = FontFamily.SansSerif,
            fontWeight = FontWeight.Bold,
            fontSize = 20.sp
        ),
        bodyMedium = TextStyle(
            fontSize = 14.sp
        )
    )

    MaterialTheme(
        colorScheme = colorScheme,
        typography = typography,
        content = content
    )
}

```

이렇게 설정한 테마는 `setContent` 블록 내부에서 전체 앱에 적용됩니다.

```kotlin
setContent {
    FestaBookTheme {
        FAQScreen(...)
    }
}
```

이를 통해 **일관된 색상, 폰트, 컴포넌트 스타일**을 유지할 수 있고, 팀 내 UI 디자인 변경이 발생해도 중앙에서 쉽게 수정 가능합니다.

## **6. 결론 및 향후 방향**
이번 점진적 마이그레이션을 통해 Jetpack Compose를 도입하는 과정은 단순히 UI를 바꾸는 것이 아니라, 기존 구조와 화면 설계 방식을 점진적으로 개선하는 경험이었습니다. ComposeView와 AndroidView를 활용하여 기존 XML 기반 화면과 Compose를 공존시키면서도, 안정적으로 선언형 UI를 적용할 수 있음을 확인했습니다.
공통 UI 컴포넌트와 테마를 통합함으로써 코드의 재사용성과 일관성을 확보할 수 있었습니다.
