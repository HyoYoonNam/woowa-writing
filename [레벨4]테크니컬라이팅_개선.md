# 📑 안드로이드 XML에서 Jetpack Compose로 점진적 마이그레이션 가이드

---

## **0. 서론: Compose 도입 배경과 전략**

우아한테크코스 7기에서는 우선 XML 기반의 안드로이드 화면 작성을 공부했습니다.

이 과정에서 뷰 바인딩(View Binding), 데이터 바인딩(Data Binding) 등의 기술을 활용하여 UI를 제어하는 경험을 쌓았습니다.

저 또한 XML 기반 UI 학습을 통해 화면 구성과 상태 관리를 익혔습니다.

이후 **레벨 4에서 Jetpack Compose를 학습**하게 되었고, 미션 과제로 기존 XML 프로젝트의 일부 화면을 Compose로 전환하는 경험을 하게 되었습니다.

이 과정에서 **작은 단위부터 Compose를 도입하고 점진적으로 화면을 전환**하는 경험을 통해, 점진적 마이그레이션 전략의 효용을 체감할 수 있었습니다.

따라서 본 글의 주제는 **XML 기반 안드로이드 프로젝트를 Compose로 점진적 전환하는 방법**으로 설정하게 되었습니다.

---

### **왜 Compose인가?**

Jetpack Compose는 2019년 Google I/O에서 처음 공개된 이후, 안드로이드 UI 개발의 패러다임을 근본적으로 바꿨습니다.

이전의 XML 방식은 View를 직접 조작해야 했기 때문에, 복잡한 화면일수록 **상태 불일치나 갱신의 어려움**이 발생했습니다.

반면 Compose는 다음과 같은 장점을 제공합니다.

- **상태(State)** 를 기반으로 UI를 자동 재구성 (Recomposition)
- **더 적은 코드로 일관된 UI 표현**
- **테스트·유지보수 용이성** 향상

```kotlin
// XML 기반 명령형 UI
val textView: TextView = findViewById(R.id.textView)
textView.text = "안녕하세요"

// Compose 선언형 UI
@Composable
fun Greeting(name: String) {
    Text(text = "안녕하세요, $name")
}

```

---

### **점진적 마이그레이션 전략**

Compose 전환은 **한 번에 전체를 바꾸는 것이 아니라**, 다음과 같은 단계로 점진적으로 진행하는 것이 바람직합니다.

> 🔹 Compose 마이그레이션 단계 요약
>
> 1. **소규모 컴포넌트**부터 Compose로 구현
> 2. **ComposeView**를 이용해 기존 XML 화면 내 삽입
> 3. **AndroidView**로 기존 View 재활용
> 4. 신규 화면은 **Compose로 완전 전환**

이 가이드는 위의 흐름에 따라 실제 프로젝트(festabook)에서의 적용 사례와 코드를 단계별로 설명합니다.

## **1. 개발 환경 준비 및 기존 UI 분석**

### **Compose 환경 설정**

Compose를 도입하기 위해서는 Gradle 설정에서 Compose를 활성화해야 합니다.

```kotlin
android {
    buildFeatures {
        compose = true // Compose 기능 활성화
    }
    composeOptions {
        kotlinCompilerExtensionVersion = "version_name"
    }
}

dependencies {
    // Compose BOM으로 버전 일괄 관리
    implementation(platform("androidx.compose:compose-bom:2025.10.00"))

    // 핵심 Compose 라이브러리
    implementation("androidx.compose.ui:ui")
    implementation("androidx.compose.material3:material3")
    implementation("androidx.compose.ui:ui-tooling-preview")
    debugImplementation("androidx.compose.ui:ui-tooling")
    implementation("androidx.activity:activity-compose:1.9.0")
}

```

> 💡 Tip:
>
>
> Compose BOM(Bill of Materials)을 사용하면 라이브러리 버전을 통합적으로 관리할 수 있습니다.
>

---

### **기존 UI 분석 기준**

어떤 화면부터 Compose로 전환할지는 다음 세 가지 기준으로 판단합니다.

| 분석 항목 | 설명 | 예시 |
| --- | --- | --- |
| **UI 복잡도** | 중첩 뷰, 커스텀 레이아웃 등 구조적 복잡성 | ConstraintLayout, RecyclerView |
| **상태 의존성** | LiveData, ViewModel에 의존하는 정도 | 상태가 잦은 화면 |
| **재사용성** | 여러 화면에서 반복되는 UI | 카드, 버튼, 리스트 아이템 |

> ✅ 전략 포인트:
>
>
> 단순하고 독립적인 구성요소부터 Compose로 옮기는 것이 가장 안전합니다.
>

festabook 프로젝트에서는 ‘공지사항’과 ‘FAQ’ 카드가 동일한 구조를 공유했기 때문에,

**재사용성이 높고 상태 의존성이 낮은 UI**로 판단되어 첫 전환 대상으로 선정했습니다.

---

## **2. 컴포넌트 단위 전환: 작은 UI부터 시작하기**

### **2.1 전환 대상 선정**

Compose 전환 초기에는 **재사용성과 독립성이 높은 컴포넌트**가 가장 적합합니다.

| 선정 기준 | 설명 | 예시 |
| --- | --- | --- |
| **재사용성** | 여러 화면에서 반복되는 UI | 카드, 버튼 |
| **독립성** | 비즈니스 로직 의존도 낮음 | 단순 정보 표시용 View |
| **상태 단순성** | 클릭, 확장 등 단순 상태 | FAQ, 공지 아이템 |

---

### **2.2 XML → Compose 변환 예시**

```kotlin
@Composable
fun NewsItem(
    title: String,
    description: String,
    isExpanded: Boolean,
    onClick: () -> Unit,
) {
    Column(
        modifier = Modifier
            .fillMaxWidth()
            .clickable { onClick() }
            .animateContentSize()
            .padding(16.dp)
    ) {
        Text(text = title, fontSize = 14.sp)
        if (isExpanded) {
            Spacer(Modifier.height(8.dp))
            Text(text = description)
        }
    }
}

```

> 💬 의도:
>
>
> 단순한 클릭/확장형 아이템을 Compose로 표현한 예시입니다.
>
> `animateContentSize()`로 부드러운 확장 애니메이션을 간결하게 구현할 수 있습니다.
>

---

### **2.3 기존 XML 화면에 Compose 삽입하기**

ComposeView를 활용하면 **기존 XML 화면 일부 영역만 Compose로 교체**할 수 있습니다.

**(1) XML 레이아웃**

```xml
<androidx.compose.ui.platform.ComposeViewandroid:id="@+id/newsItemCompose"
    android:layout_width="match_parent"
    android:layout_height="wrap_content" />

```

**(2) Fragment/Activity에서 Compose 설정**

```kotlin
// 기존 ViewModel 상태를 그대로 사용하며 Compose UI로 연결
binding.newsItemCompose.setContent {
    NewsItem(
        title = "공지사항",
        description = "서비스가 업데이트되었습니다.",
        isExpanded = viewModel.isExpanded,
        onClick = { viewModel.toggleExpand() }
    )
}

```

> ✅ 핵심:
>
>
> 기존 로직은 그대로 두고 UI만 Compose로 교체할 수 있습니다.
>
> 안정적으로 Compose를 테스트하기 좋은 시작점입니다.
>

---

### **2.4 화면 단위로 ComposeView 구성**

Compose 도입이 익숙해지면, **Fragment 전체를 Compose로 구성**할 수도 있습니다.

```kotlin
class FAQFragment : Fragment() {
    private val viewModel: NewsViewModel by viewModels()

    override fun onCreateView(
        inflater: LayoutInflater,
        container: ViewGroup?,
        savedInstanceState: Bundle?,
    ): View =
        ComposeView(requireContext()).apply {
            setViewCompositionStrategy(ViewCompositionStrategy.DisposeOnViewTreeLifecycleDestroyed)
            setContent {
                FAQScreen(
                    uiState = viewModel.faqUiState,
                    onFaqClick = { item -> viewModel.onClick(item) }
                )
            }
        }
}

```

> 💡 Note:
>
>
> XML 기반 Activity나 네비게이션 구조를 유지하면서도 화면 단위 전환이 가능합니다.
>

## **3. AndroidView로 기존 View 재사용하기**

Compose로 화면을 옮기더라도, 기존 View를 완전히 제거하기 어려운 경우가 있습니다.

특히 `MapView`, `WebView`처럼 Compose 버전이 없거나 외부 SDK를 사용하는 경우가 대표적입니다.

이때 `AndroidView`를 활용하면 Compose 내부에서 기존 View를 그대로 재사용할 수 있습니다.

```kotlin
@Composable
fun MapContainerView() {
    AndroidView(
        factory = { context ->
            MapView(context).apply {
                getMapAsync { map ->
                    map.uiSettings.isZoomControlEnabled = false
                }
            }
        },
        update = { mapView ->
            // 상태 변경 시 UI 업데이트
        }
    )
}

```

> ✅ 핵심 포인트:
>
>
> Compose와 기존 View를 **양방향으로 연결**할 수 있습니다.
>
> `factory`는 초기화, `update`는 상태 변경 시 호출됩니다.
>

> 💡 추가 Tip:
>
>
> Compose 1.5+에서는 `AndroidViewBinding`을 통해 **XML 레이아웃 전체를 Compose 내에서 재사용**할 수 있습니다.
>
> ```kotlin
> AndroidViewBinding(ItemMapBinding::inflate) { binding ->
>     binding.mapView.getMapAsync { ... }
> }
> 
> ```
>

## **4. 신규 화면은 Compose로 완전 전환하기**

점진적 전환이 안정화되면, 신규 화면은 **처음부터 Compose로 작성**합니다.

이때는 XML 레이아웃이 전혀 필요하지 않습니다.

```kotlin
class FAQActivity : ComponentActivity() {
    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        setContent {
            FestaBookTheme {
                FAQScreen()
            }
        }
    }
}

```

> ✅ 이점:
>
>
> Compose Navigation, ViewModel 연계, UI Test 등을 통합적으로 관리할 수 있습니다.
>

## **5. 공통 컴포넌트화 및 테마 일원화**

### **5.1 공통 UI 컴포넌트 추출**

Compose의 Composable 함수는 **UI 컴포넌트로 쉽게 모듈화**할 수 있습니다.

```kotlin
@Composable
fun FestaButton(
    text: String,
    onClick: () -> Unit,
    modifier: Modifier = Modifier,
    enabled: Boolean = true
) {
    Button(
        onClick = onClick,
        enabled = enabled,
        modifier = modifier
            .fillMaxWidth()
            .height(48.dp)
            .clip(RoundedCornerShape(12.dp))
    ) {
        Text(
            text = text,
            style = MaterialTheme.typography.labelLarge,
            color = MaterialTheme.colorScheme.onPrimary
        )
    }
}

```

> 💡 관리 팁:
>
>
> 공통 UI는 `ui/component` 패키지에 모아 관리하면 화면 단위 Composable(`Screen`)과 명확히 구분할 수 있습니다.
>

---

### **5.2 MaterialTheme으로 디자인 일원화**

Compose는 `MaterialTheme`을 통해 색상, 폰트, 간격 등을 **중앙 관리**할 수 있습니다.

```kotlin
@Composable
fun FestaBookTheme(content: @Composable () -> Unit) {
    val colorScheme = lightColorScheme(
        primary = Color(0xFF0061A4),
        secondary = Color(0xFF5E92F3),
        background = Color.White,
        onPrimary = Color.White,
        onBackground = Color.Black
    )

    val typography = Typography(
        titleLarge = TextStyle(
            fontFamily = FontFamily.SansSerif,
            fontWeight = FontWeight.Bold,
            fontSize = 20.sp
        ),
        bodyMedium = TextStyle(fontSize = 14.sp)
    )

    MaterialTheme(colorScheme = colorScheme, typography = typography, content = content)
}

```

> ✅ 효과:
>
>
> 전체 앱의 UI를 일관성 있게 유지하며, 테마 변경 시 수정 범위를 최소화할 수 있습니다.
>

## **6. 결론 및 향후 방향**

이번 점진적 마이그레이션을 통해 Jetpack Compose를 도입하는 과정은

단순히 UI 프레임워크를 바꾸는 것이 아니라 **UI 구조와 설계 방식 자체를 개선하는 경험**이었습니다.

ComposeView와 AndroidView를 활용하면 기존 XML 화면과 Compose를 공존시킬 수 있으며,

공통 UI 컴포넌트와 테마를 통합함으로써 코드의 재사용성과 일관성을 높일 수 있었습니다.

festabook 팀은 앞으로 남은 XML 화면을 순차적으로 Compose로 전환하고,

Compose Navigation, Animation 등 다양한 기능을 적용하며

**프로젝트 전반의 유지보수성과 개발 생산성**을 높일 계획입니다.
