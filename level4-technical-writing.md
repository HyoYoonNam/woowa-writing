### **1. NoSQL의 등장 배경**

NoSQL은 'Not Only SQL'의 약자로, 단순히 SQL을 사용하지 않는 데이터베이스를 넘어, 전통적인 관계형 데이터베이스(RDB)와 다른 방식으로 데이터를 저장하는 모든 비관계형 데이터베이스를 지칭합니다. NoSQL이 등장하고 빠르게 확산된 배경에는 현대 IT 환경의 근본적인 변화가 있습니다.

### **1-1. 데이터 빅뱅과 확장성의 요구**

Google, Amazon, Facebook과 같은 거대 웹 서비스의 등장으로 생성되고 처리해야 할 데이터의 양이 기하급수적으로 증가했습니다. 기존의 단일 서버 기반 RDB 시스템은 이러한 대용량 비정형 데이터를 처리하는 데 점차 한계를 드러내기 시작했습니다.

### **1-2. 서버 확장 방식의 진화: 스케일 업 vs 스케일 아웃**

데이터 처리 요구량이 증가함에 따라 서버의 성능을 확장하는 방식에 대한 고민이 시작되었습니다. 확장 방식은 크게 두 가지로 나뉩니다.

- **스케일 업 (Scale-Up, 수직적 확장)** 🚀
  기존 서버 자체의 하드웨어 사양(CPU, RAM 등)을 더 좋은 부품으로 교체하여 성능을 높이는 방식입니다. 직관적이지만 고사양 부품은 매우 비싸고, 물리적인 성능 향상에도 명백한 한계가 존재합니다.
- **스케일 아웃 (Scale-Out, 수평적 확장)** 💻💻💻
  여러 대의 저사양 서버를 연결하여 클러스터를 구성하고, 데이터를 분산하여 병렬로 처리하는 방식입니다. 상대적으로 저렴한 비용으로 시스템을 유연하게 확장할 수 있어 대규모 분산 시스템의 표준적인 확장 방식으로 자리 잡았습니다.

### **1-3. 관계형 데이터베이스의 수평적 확장 한계**

문제는 대부분의 관계형 데이터베이스가 **단일 서버 환경을 가정하고 설계**되었다는 점입니다. RDB의 핵심은 정규화를 통해 데이터를 여러 테이블에 나누어 저장하고, 필요할 때 **JOIN** 연산을 통해 테이블들을 연결하여 원하는 정보를 조합하는 것입니다.

하지만 데이터를 여러 서버에 분산하는 **수평적 확장 환경에서는 이 '관계'를 유지하는 것이 매우 복잡하고 어렵습니다**. 여러 서버에 흩어진 테이블들을 JOIN하는 작업은 극심한 성능 저하를 유발하며, RDB가 스케일 아웃 방식에 쉽게 적응하지 못하는 근본적인 원인이 되었습니다. 이러한 한계를 극복하기 위해 탄생한 것이 바로 NoSQL 데이터베이스입니다.

### **2. MongoDB 소개: 도큐먼트 지향 데이터베이스**

MongoDB는 수많은 NoSQL 데이터베이스 중 가장 널리 사용되는 **도큐먼트 지향(Document-Oriented) 데이터베이스**입니다.

### **2-1. 도큐먼트와 컬렉션: RDB와의 개념 비교**

MongoDB의 데이터 구조는 RDB와 매우 다릅니다.

- **테이블(Table) → 컬렉션(Collection)**
- **행(Row) → 도큐먼트(Document)**

RDB가 정해진 스키마를 가진 테이블의 행에 데이터를 저장하는 반면, MongoDB는 유연한 구조를 가진 **도큐먼트**들의 모음인 **컬렉션**에 데이터를 저장합니다.

### **2-2. BSON: JSON을 넘어선 데이터 형식**

MongoDB는 데이터를 **JSON(JavaScript Object Notation)과 유사한 BSON(Binary JSON) 형식으로 저장**합니다. BSON은 JSON의 유연성과 가독성을 유지하면서, 바이너리(Binary) 형식으로 인코딩되어 데이터 처리 속도가 더 빠르고, 추가적인 데이터 타입(날짜, 바이너리 데이터 등)을 지원하는 장점이 있습니다.

---

### **3. MongoDB 데이터 모델링**

RDB에서 MongoDB로 전환할 때 가장 중요한 것은 **데이터 모델링 방식의 차이**를 이해하는 것입니다. RDB가 데이터 중복을 피하기 위해 정규화(Normalization)를 지향한다면, MongoDB는 **조회 성능**을 높이기 위해 의도적으로 비정규화(Denormalization)를 활용하는 경우가 많습니다.

### **3-1. 내장(Embedding) 모델: JOIN을 제거하는 가장 확실한 방법**

연관된 데이터를 하나의 도큐먼트 안에 모두 포함시키는 방식입니다. 이는 MongoDB의 가장 기본적이고 강력한 모델링 기법입니다.

- **예시**: '게시글'과 '댓글'
  RDB에서는 `posts`와 `comments` 테이블을 분리하지만, MongoDB에서는 하나의 게시글 도큐먼트 안에 `comments`를 배열로 내장할 수 있습니다.

    ```json
    {
      "_id": ObjectId("6748a1b2c5a1b3a1c2d3e4f5"),
      "title": "MongoDB 도입기",
      "content": "블로그 글 내용",
      "author_name": "개발자A",
      "tags": ["백엔드", "DB", "NoSQL"],
      "comments": [
        {
          "comment_id": ObjectId("a1b2c3d4e5f6a7b8c9d0e1f2"),
          "author_name": "독자B",
          "text": "첫 글 축하드립니다! 앞으로 좋은 내용 기대할게요."
        },
        {
          "comment_id": ObjectId("b2c3d4e5f6a7b8c9d0e1f2a3"),
          "author_name": "개발자C",
          "text": "저도 몽고DB로 프로젝트 중인데 반갑습니다!"
        }
      ]
    }
    ```


- **장점**: 관련된 모든 데이터를 한 번의 읽기 작업으로 가져올 수 있어 조회 성능이 매우 빠릅니다. 데이터의 원자성(Atomicity)을 하나의 도큐먼트 단위로 보장하기도 쉽습니다.
- **단점**: 내장된 데이터가 계속 증가하면 도큐먼트의 크기가 너무 커질 수 있습니다 (최대 16MB). 또한, 여러 도큐먼트에 동일한 정보가 중복 저장될 수 있습니다.

### **3-2. 참조(Referencing) 모델: 정규화가 필요할 때**

RDB의 외래 키(Foreign Key)처럼, 한 도큐먼트에서 다른 도큐먼트의 `_id` 값을 참조하는 방식입니다.

- **예시**: '상품'과 '주문'
  하나의 상품이 수천, 수만 개의 주문에 포함될 수 있습니다. 이때 모든 주문 정보를 상품 도큐먼트에 내장하는 것은 비효율적이므로, `orders` 컬렉션에서 `product_id`를 참조하도록 설계합니다.

    ```json
    // products 컬렉션
    {
      "_id": "product123",
      "name": "Real MySQL 2.0 1편",
      "price": 15000
    }
    
    // orders 컬렉션
    {
      "_id": "order456",
      "product_id": "product123", // 상품 참조
      "quantity": 2,
      "order_date": ISODate("2025-10-14T11:30:00Z")
    }
    ```

- **장점**: 데이터 중복을 피할 수 있고, 도큐먼트 크기가 커지는 것을 방지합니다.
- **단점**: RDB의 `JOIN`처럼, 관련된 데이터를 가져오기 위해 애플리케이션에서 여러 번의 쿼리를 실행하거나 MongoDB의 `$lookup` (집계 연산자)을 사용해야 하므로 조회 성능이 저하될 수 있습니다.

### **3-3. 어떤 모델을 선택할 것인가?**

모델 선택은 정답이 없으며, 데이터의 관계와 애플리케이션의 읽기/쓰기 패턴에 따라 결정해야 합니다.

- **내장(Embedding) 모델이 적합한 경우**:
    - 데이터 간의 관계가 **'Contains'(포함)** 관계일 때 (예: 게시글과 댓글)
    - 읽기 성능이 매우 중요할 때
    - 데이터가 함께 조회되는 경우가 대부분일 때
- **참조(Referencing) 모델이 적합한 경우**:
    - 내장 시 도큐먼트 크기가 16MB를 초과할 위험이 있을 때
    - 참조되는 데이터가 자주 수정되어 데이터 정합성 유지가 중요할 때
    - 데이터 간의 관계가 복잡하고 다대다(N:M) 관계를 표현해야 할 때

### **4. MongoDB 쿼리**

MongoDB는 JavaScript 객체 형식을 기반으로 한 직관적인 쿼리 언어를 사용합니다. RDB의 SQL과 비교하며 CRUD(Create, Read, Update, Delete) 작업을 살펴보겠습니다.

### **4-1. 기본 CRUD 쿼리: RDB와 비교**

- 생성

    ```sql
    INSERT INTO items (name, price) VALUES ('상품A', 1000);
    ```

    ```javascript
    db.items.insertOne({ name: "상품A", price: 1000 })
    ```

- 조회

    ```sql
    SELECT * FROM users WHERE name = '개발자A';
    ```

    ```javascript
    db.users.find({ name: "개발자A" })
    ```

- 수정

    ```sql
    UPDATE users SET email = 'new@email.com' WHERE name = '개발자A';
    ```

    ```javascript
    db.users.updateOne({ name: "개발자A" }, { $set: { email: "new@email.com" } })
    ```

- 삭제

    ```sql
    DELETE FROM users WHERE status = 'inactive';
    ```

    ```javascript
    db.users.deleteMany({ status: "inactive" })
    ```


### **4-2. 고급 쿼리: Aggregation Framework 소개**

RDB의 `GROUP BY`, `JOIN`, 복잡한 서브쿼리처럼 여러 단계의 데이터 처리가 필요할 때, MongoDB는 **Aggregation Framework**를 사용합니다. 이는 데이터 처리 단계를 파이프라인(Pipeline) 형태로 연결하여 복잡한 연산을 수행하는 강력한 기능입니다.

- **예시**: `orders` 컬렉션에서 상태가 'A'인 상품들만 골라, 고객 ID별로 주문 금액의 합계를 계산하는 경우

    ```javascript
    db.orders.aggregate([
      // 1단계: $match (필터링) - WHERE 절과 유사
      { $match: { status: "A" } },
    
      // 2단계: $group (그룹화) - GROUP BY 절과 유사
      { $group: {
          _id: "$cust_id",  // 고객 ID를 기준으로 그룹화
          total: { $sum: "$amount" } // amount 필드의 합계를 total 이라는 새로운 필드로 생성
      }}
    ])
    ```

### **5. 분산 시스템 이론: CAP과 BASE**

분산 데이터베이스를 이해하기 위해서는 **CAP 이론**과 **BASE 원칙**을 알아야 합니다. 이 이론들은 분산 시스템이 가질 수 있는 특성과 그에 따른 트레이드오프를 설명합니다.

### **5-1. CAP 이론이란?**

CAP 이론은 분산 데이터베이스 시스템은 다음 세 가지 특성 중 **최대 두 가지만을 동시에 보장**할 수 있다는 이론입니다.

- **일관성 (Consistency)**: 모든 노드가 항상 동일한 데이터를 보여주는 것을 보장합니다. 즉, 쓰기 작업이 성공하면 이후의 모든 읽기 작업은 그 최신 데이터를 반환해야 합니다.
- **가용성 (Availability)**: 모든 요청(읽기/쓰기)에 대해 시스템이 항상 응답하는 것을 보장합니다. 즉, 일부 노드에 장애가 발생하더라도 서비스는 중단되지 않아야 합니다.
- **분할 허용성 (Partition Tolerance)**: 네트워크 문제로 인해 노드 간의 통신이 끊어지더라도(네트워크 분할), 시스템이 계속 동작하는 것을 보장합니다.

현대의 분산 시스템은 네트워크 장애를 피할 수 없으므로, **분할 허용성(P)은 반드시 가져가야 할 기본 전제**입니다. 따라서 개발자는 **일관성(C)과 가용성(A) 사이에서 무엇을 더 우선할지 선택**해야 합니다.

### **5-2. MongoDB의 선택: 유연한 CP 시스템**

MongoDB는 기본적으로 일관성(C)과 분할 허용성(P)을 우선하는 CP 시스템으로 분류됩니다. 레플리카 셋 아키텍처에서 네트워크 분할이 발생하여 Primary 노드와 Secondary 노드 간의 통신이 끊어지면, 데이터 불일치를 막기 위해 새로운 Primary를 선출하는 과정에서 일시적으로 쓰기 요청에 대한 응답을 중단할 수 있습니다. 이는 데이터의 정합성을 최우선으로 고려한 설계입니다.

하지만 MongoDB는 쓰기/읽기 설정(Write Concern/Read Preference)을 조절하여 이 트레이드오프를 유연하게 관리할 수 있습니다. 예를 들어, 쓰기 작업을 과반수 노드의 확인(majority)이 아닌 단일 노드(Primary)의 확인만 받고 반환하도록 설정하면 일관성을 낮추는 대신 응답 속도(가용성)를 높일 수 있습니다.

### **5-3. BASE 원칙: NoSQL이 일관성을 다루는 방식**

많은 NoSQL 시스템은 RDB의 엄격한 일관성(ACID) 대신 **BASE 원칙**을 따릅니다.

- **Basically Available (기본적 가용성)**: 시스템은 항상 가용한 상태를 유지합니다. (CAP의 가용성과 유사)
- **Soft State (부드러운 상태)**: 시스템의 상태는 외부의 개입이 없어도 시간이 지남에 따라 변할 수 있습니다. 예를 들어, 노드 간 동기화 과정에서 같은 데이터에 대한 조회 결과가 시간에 따라 다를 수 있습니다.
- **Eventually Consistent (최종적 일관성)**: 데이터의 일관성이 즉시 보장되지는 않지만, **결국에는(Eventually)** 모든 노드의 데이터가 일관된 상태로 동기화되는 것을 보장합니다.

BASE는 가용성을 극대화하기 위해 일관성을 약간 희생하는 모델로, 소셜 미디어 피드처럼 약간의 데이터 지연이 허용되는 서비스에 적합합니다. MongoDB는 기본적으로 CP 시스템이지만, 읽기 설정(Read Preference)을 조정하여 최종적 일관성 모델로도 동작할 수 있는 유연성을 제공합니다.

### **6. MongoDB 고가용성과 수평적 확장**

MongoDB의 가장 큰 강점은 대규모 분산 환경을 위해 설계된 **고가용성**(High Availability)과 **수평적 확장성**(Horizontal Scalability)입니다. 이 두 가지 특징은 각각 **복제**(Replication)와 **샤딩**(Sharding)이라는 핵심 아키텍처를 통해 구현됩니다.

![sharded-cluster-production-architecture](https://github.com/user-attachments/assets/6136f422-648f-474e-a1dc-7ee8d9e374f9)

### **6-1. 복제(Replication) 심층 분석: 데이터 안전성의 비밀**

- **복제**(Replication)란 여러 서버에 **동일한 데이터의 복사본**을 유지하여 데이터의 안정성과 가용성을 높이는 기술입니다. 하나의 서버에 장애가 발생하더라도 다른 서버에 복제된 데이터가 존재하므로, 서비스 중단 없이 데이터를 안전하게 보존하고 서비스를 지속할 수 있습니다.

MongoDB에서는 이 복제 기능을 **레플리카 셋**(Replica Set)이라는 개념으로 구현합니다.

- **레플리카 셋의 구성 요소**
    - **Primary (프라이머리)**: 레플리카 셋에서 **오직 하나만 존재**하며, 모든 데이터 **쓰기(Write) 작업을 처리**하는 유일한 노드입니다. Primary에서 발생한 모든 데이터 변경 사항은 Oplog라는 특별한 로그에 기록됩니다.
    - **Secondary (세컨더리)**: Primary의 Oplog를 실시간으로 복제하여 **데이터 복사본을 유지**하는 노드입니다. 기본적으로 데이터 읽기(Read) 작업을 처리하여 Primary의 읽기 부하를 분산시키는 역할을 할 수 있습니다. Primary에 장애가 발생하면 새로운 Primary가 되기 위한 선거에 참여합니다.
    - **Arbiter (아비터 - 선택 사항)**: 데이터는 저장하지 않지만, Primary 선거 과정에서 **투표권만 행사**하는 특별한 노드입니다. 노드 개수를 홀수로 맞춰야 할 때 최소한의 리소스로 쿼럼(Quorum, 의사결정에 필요한 최소 투표수)을 구성하기 위해 사용됩니다.
- **데이터 동기화 과정: Oplog의 역할**
  Secondary 노드는 어떻게 Primary와 데이터를 똑같이 유지할까요? 비밀은 Oplog(Operations Log)에 있습니다.
    1. 애플리케이션이 Primary에 쓰기 요청(insert, update, delete)을 보냅니다.
    2. Primary는 데이터를 변경하고, 이 **변경 작업에 대한 상세 내용**을 자신의 **Oplog**에 기록합니다. Oplog는 '어떤 데이터를 어떻게 변경했다'는 모든 작업 내역을 순서대로 저장하는 일종의 트랜잭션 로그입니다.
    3. 모든 Secondary 노드들은 주기적으로 Primary의 Oplog를 확인하며, 자신이 마지막으로 복제한 이후에 새로운 로그가 있는지 감시합니다.
    4. 새로운 로그가 발견되면, Secondary는 그 로그를 가져와 자신에게 똑같이 재현(replay)함으로써 데이터를 동기화합니다.
- **자동 장애 복구 (Automatic Failover) 메커니즘**
  레플리카 셋의 가장 중요한 기능은 Primary 노드에 장애가 발생했을 때 서비스 중단을 최소화하며 자동으로 복구하는 것입니다.
    1. **상태 확인 (Heartbeat)**: 레플리카 셋의 모든 노드들은 2초마다 서로에게 **하트비트(Heartbeat)**라는 작은 신호를 보내 "나 살아있어"라고 알립니다.
    2. **장애 감지 (Timeout)**: 만약 특정 노드가 다른 노드로부터 10초 이상 하트비트 신호를 받지 못하면, 그 노드에 장애가 발생했다고 간주합니다.
    3. **선거 시작 (Election)**: Primary에 장애가 감지되면, 남아있는 Secondary 노드들 사이에서 즉시 새로운 Primary를 선출하기 위한 **선거**가 시작됩니다.
    4. **투표 및 당선**:
        - 가장 최신 Oplog를 가진, 즉 가장 최신 데이터를 가진 Secondary 노드가 **후보**로 출마합니다.
        - 다른 노드들은 후보자가 자신보다 더 최신 데이터를 가졌는지 확인하고 **투표**합니다.
        - 후보자가 레플리카 셋의 **과반수**로부터 표를 받으면 새로운 **Primary로 승격**됩니다.
    5. **서비스 재개**: 새로운 Primary가 선출되면, 애플리케이션은 새로운 Primary에 쓰기 요청을 보내며 서비스는 중단 없이 계속됩니다.

### **6-2. 샤딩(Sharding) 심층 분석: 무한한 수평적 확장의 비밀**

- **샤딩**(Sharding)은 하나의 거대한 데이터셋을 **샤드**(Shard)라고 하는 여러 개의 작은 조각으로 나누어, 다수의 서버에 **분산 저장**하는 기술입니다. 복제가 데이터의 *안전성*을 위한 기술이라면, 샤딩은 **대용량 데이터 저장**과 **쓰기 성능 확장**을 위한 기술입니다.
- **샤드 클러스터의 3대 구성 요소**
    - **샤드 (Shard)**: 분산된 데이터가 실제로 저장되는 서버입니다. 단일 장애 지점(SPOF)을 방지하고 데이터 안정성을 높이기 위해, **각각의 샤드는 그 자체로 레플리카 셋으로 구성**됩니다. 즉, 분산된 데이터 조각 하나하나가 고가용성을 갖추게 됩니다.
    - **Mongos (라우터)**: 애플리케이션과 샤드 클러스터 사이의 **게이트웨이** 역할을 하는 경량 라우터입니다. 애플리케이션으로부터 모든 쿼리를 받아서, Config 서버의 메타데이터를 참조하여 **어떤 샤드로 쿼리를 보내야 할지 결정하고 중계**합니다.
    - **Config 서버 (설정 서버)**: 샤드 클러스터의 **두뇌이자 지도**입니다. 어떤 데이터 조각(Chunk)이 어느 샤드에 저장되어 있는지에 대한 **메타데이터**를 모두 저장하고 관리합니다. Config 서버 자체의 안정성이 매우 중요하므로, Config 서버 역시 레플리카 셋으로 구성됩니다.
- **샤딩의 핵심: 샤드 키 (Shard Key)와 데이터 분산 과정**
  MongoDB는 어떻게 데이터를 여러 샤드에 골고루 분산시킬까요? 그 비밀은 **샤드 키**(Shard Key)에 있습니다.
    1. **샤드 키 선택**: 샤딩을 적용할 컬렉션에서 하나의 필드(또는 여러 필드의 조합)를 **샤드 키**로 지정합니다. 이 샤드 키의 값을 기준으로 데이터가 분산되므로, 어떤 필드를 샤드 키로 선택하는지가 전체 클러스터의 성능을 좌우할 만큼 매우 중요합니다.
    2. **청크(Chunk) 분할**: MongoDB는 샤드 키의 값 범위를 기준으로 데이터를 청크(Chunk)라는 논리적인 단위로 분할합니다. 각 청크는 특정 범위의 샤드 키 값을 가진 도큐먼트들의 집합입니다. (예: 사용자 ID 1~1000은 청크 A, 1001~2000은 청크 B)
    3. **청크 분산**: **Balancer**라는 MongoDB의 백그라운드 프로세스가 주기적으로 각 샤드의 청크 개수를 모니터링합니다. 만약 특정 샤드에 청크가 몰려있다고 판단되면, Balancer는 해당 샤드의 청크를 다른 샤드로 **자동으로 이동시켜** 데이터가 클러스터 전체에 균등하게 분산되도록 유지합니다.
- **쿼리 처리 흐름**
    1. 애플리케이션이 **Mongos**로 쿼리를 보냅니다. (예: `name`이 'apple'인 과일 찾기)
    2. **Mongos**는 **Config 서버**에 "name 필드가 'apple'인 데이터는 어느 샤드에 있나요?"라고 물어봅니다.
    3. **Config 서버**는 메타데이터를 참조하여 "그 데이터는 샤드 1에 있습니다"라고 알려줍니다.
    4. **Mongos**는 해당 쿼리를 **샤드 1**로 전달합니다.
    5. **샤드 1**은 쿼리를 처리하여 결과를 **Mongos**로 반환하고, **Mongos**는 최종 결과를 애플리케이션에 전달합니다.

이처럼 샤딩은 복제와 유기적으로 결합하여, 대규모 데이터를 여러 서버에 안전하게 분산 저장하고 처리함으로써 MongoDB가 거의 무한에 가까운 수평적 확장을 이룰 수 있게 하는 핵심 기술입니다.

### **7. MongoDB의 추가 기능 및 활용 사례**

MongoDB는 기본적인 데이터 저장 기능 외에도 현대적인 애플리케이션 개발에 유용한 강력한 추가 기능들을 제공합니다.

### **7-1. 주요 추가 기능**

- **공간 데이터 처리 (Geospatial Queries)**: `2dsphere` 인덱스를 통해 위도/경도 기반의 공간 데이터를 매우 효율적으로 처리할 수 있습니다. "내 주변 1km 내 장소 찾기"와 같은 위치 기반 서비스(LBS)를 데이터베이스 수준에서 강력하게 지원합니다.
- **TTL 인덱스 (Time-To-Live)**: 데이터의 유효기간을 설정하여, 지정된 시간이 지나면 데이터베이스가 자동으로 해당 도큐먼트를 삭제하는 기능입니다. 세션 정보, 임시 인증 토큰, 기간 한정 로그 데이터 등을 관리하는 데 매우 유용합니다.
- **복합 인덱스 (Compound Index)**: 두 개 이상의 필드를 결합하여 하나의 인덱스를 생성하는 기능으로, 다중 조건 검색 쿼리의 성능을 최적화할 수 있습니다.

### **7-2. 국내외 기업 활용 사례**

MongoDB는 대용량 데이터를 다루거나, 데이터 간의 연관도가 낮고, 빠른 개발 속도가 중요한 서비스에서 널리 활용됩니다.

- 우아한***: 조회와 쓰기 요청의 특징이 다른 점에 착안하여, CQRS(Command and Query Responsibility Segregation) 패턴을 적용했습니다. 복잡한 조회 요청을 처리하는 조회용 데이터베이스로 MongoDB를 활용하여 시스템의 성능과 안정성을 높였습니다.
- 카**: 대규모 서비스에서 발생하는 **방대한 양의 로그 데이터를 저장하고 분석**하는 용도로 MongoDB를 사용합니다. 또한, 카**맵, 카**택시 등 위치 기반 서비스에서 MongoDB의 강력한 **공간 인덱스 기능을 적극적으로 활용**하고 있습니다.

### **8. MongoDB vs RDB**

MongoDB와 RDB는 각각의 강점이 있으며, 프로젝트의 특성에 따라 적합한 선택이 달라집니다.

### **8-1. MongoDB가 더 적합한 경우**

- **스키마 유연성이 필요할 때**: 요구사항이 자주 변하는 애자일 환경에서 마이그레이션 없이 구조를 변경할 수 있어 개발 속도가 빠릅니다.
- **비정형/반정형 데이터를 다룰 때**: SNS 콘텐츠, IoT 데이터, 로그처럼 구조가 다양한 데이터를 자연스럽게 저장할 수 있습니다.
- **대규모 수평적 확장이 필요할 때**: 샤딩을 통해 데이터를 여러 서버에 쉽게 분산할 수 있어 대용량 서비스에 유리합니다.
- **읽기 성능이 중요할 때**: 내장 모델을 활용하여 JOIN 없이 한 번의 쿼리로 데이터를 조회할 수 있습니다.
- **위치 기반 서비스**: 공간 인덱스 기능으로 지도, 배달, 모빌리티 서비스 구현에 강력합니다.

### **8-2. RDB가 더 적합한 경우**

- **복잡한 트랜잭션이 필요할 때**: 금융, 회계, ERP 등 여러 테이블에 걸친 ACID 보장이 필수적인 시스템에 적합합니다.
- **복잡한 관계와 JOIN이 중요할 때**: 다대다 관계가 많고 복잡한 JOIN 연산이 빈번한 경우 정규화된 테이블 구조가 효율적입니다.
- **데이터 무결성이 최우선일 때**: 외래 키, 제약조건 등 데이터베이스 레벨의 엄격한 검증이 필요한 경우 RDB가 유리합니다.
- **표준 SQL 생태계가 필요할 때**: 다양한 BI/분석 도구와의 통합이 중요한 경우 SQL 표준이 장점입니다.


실무에서는 두 데이터베이스를 함께 사용하는 경우도 많습니다. 예를 들어, 사용자 인증과 결제는 RDB로 일관성을 보장하고, 활동 로그와 추천 데이터는 MongoDB로 유연하게 처리하는 방식입니다.
핵심은 "어떤 것이 더 우수한가"가 아니라 "우리 프로젝트에 무엇이 더 적합한가"입니다. 데이터 특성, 조회 패턴, 확장성 요구사항을 종합적으로 고려하여 선택하는 것이 중요합니다.

---

### **9. 결론**

MongoDB는 단순히 'SQL을 사용하지 않는' 데이터베이스를 넘어, 대용량 데이터 시대의 요구에 부응하기 위해 탄생한 현대적인 데이터베이스 솔루션입니다. **유연한 도큐먼트 모델**을 통해 빠른 개발 속도를 지원하고, **레플리카 셋**과 **샤딩**을 통해 서비스의 안정성과 확장성을 보장합니다.

전통적인 관계형 데이터베이스가 여전히 강력한 분야가 있지만, 비정형 데이터 처리, 대규모 분산 환경, 민첩한 개발 프로세스가 중요한 프로젝트라면 MongoDB는 매우 강력하고 매력적인 선택지가 될 것입니다.
