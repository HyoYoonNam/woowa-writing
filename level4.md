# JPA 연관관계, 꼭 객체로 맺어야 할까?

## 개요

JPA(Java Persistence API)를 사용해 도메인을 설계할 때, 객체 간 관계를 어떻게 맺을지에 대해 고민하게 됩니다. 예를 들어, `Member`와 `Team`처럼 서로 연결된 엔티티가 있다면, `@ManyToOne` 같은 객체 참조를 통해 직접 연결하는 방식이 존재합니다. 하지만 이런 구조는 초기 설계 시점에 도메인 간 관계를 강하게 엮게 되고, 이는 나중에 유연하게 구조를 바꾸기 어렵게 만들 수 있습니다.

저는 이번에 프로젝트를 진행하며 다음과 같은 생각을 하게되었습니다.

- 정말 이 두 도메인이 항상 같이 존재해야 할까요?
- 지금은 연관 관계처럼 보이지만, 시간이 지나면서 독립적으로 바뀔 수도 있는 건 아닐까요?
- 처음부터 연관관계를 객체로 연결하지 않고, 단순히 Long ID로 간접적으로 들고 가면 안 되는 걸까요?

이 글은 이런 고민을 해보면서 얻은 정리입니다. JPA를 사용하면서 객체 참조 대신 Long ID 기반의 간접 참조 방식을 선택할 수 있는 기준, 그에 따른 장단점, 그리고 MySQL에서 외래 키(Foreign Key, FK) 유무에 따라 어떤 실행 흐름과 구조적 차이가 생기는지를 중심으로 정리해 보았습니다.

## **객체 연관관계(직접 참조) 방식**

### 개념

**객체 연관관계 방식**은 엔티티가 다른 엔티티를 **직접 객체 참조** 형태로 가지고 있도록 설계하는 방법입니다. 예를 들어 회원(Member) 엔티티가 팀(Team) 엔티티를 참조해야 한다면, 회원 엔티티 내부에 **`Team`** 타입의 필드를 두고 JPA의 **`@ManyToOne`**이나 **`@OneToMany`**와 같은 애노테이션으로 맵핑합니다.

이렇게 하면 데이터베이스에서는 외래 키 컬럼으로 연관관계가 표현되지만, 코드 상에서는 마치 객체 필드를 다루듯이 **직접 참조**할 수 있습니다. JPA/Hibernate 같은 ORM 프레임워크는 이 객체 연관관계를 데이터베이스 테이블의 외래 키로 매핑하고 조회나 저장 시에 자동으로 SQL 조인을 수행하거나 적절한 쿼리를 만들어 줍니다.

이 방식의 핵심 특징은 **객체 지향적인 접근**입니다. 개발자는 SQL 조인이나 외래 키 관리보다는 객체 그래프 탐색에 집중할 수 있습니다. 예를 들어 회원 엔티티에서 **`member.getTeam().getName()`**처럼 **객체를 통해 연관된 데이터에 바로 접근**할 수 있고, 이는 마치 메모리 상의 객체 관계를 다루는 것과 유사합니다. JPA가 이러한 객체 참조를 실제 DB상의 외래 키로 자동 변환해주므로, 개발자는 명시적으로 FK 값을 관리하거나 SQL로 조인할 필요 없이 **ORM이 제공하는 이점**을 누릴 수 있습니다. 이는 ORM을 사용하는 큰 장점 중 하나입니다

### 특징

객체 직접 참조 방식에서는 **지연 로딩과 즉시 로딩** 등의 ORM 기능을 활용하여 성능을 제어할 수 있습니다. 연관된 엔티티를 실제로 사용할 때까지 DB에서 조회하지 않는 **Lazy Loading**을 사용하면 불필요한 쿼리를 줄일 수 있고, 필요한 경우 **Fetch Join**이나 **Entity Graph**를 통해 한 번의 쿼리로 관련 데이터를 모두 가져오는 튜닝도 가능합니다. 또한 **영속성 컨텍스트**가 엔티티 객체들을 관리하기 때문에, 한 트랜잭션 내에서 동일한 팀을 참조하는 여러 회원 객체가 있다면 해당 팀 객체는 1개만 로딩되어 공유됩니다. 이를 통해 동일 객체에 대한 중복 쿼리나 데이터 불일치를 방지할 수 있습니다.

JPA를 통해 DDL을 생성하는 경우, 이러한 직접 참조 관계에 대해 **자동으로 외래 키 제약**을 생성하는 것이 일반적입니다. 예를 들어 **`@ManyToOne`**으로 맵핑된 **`team`** 필드는 데이터베이스에 **`team_id`** 컬럼을 만들고 대상 테이블(팀 테이블)의 기본 키(PK)에 대한 외래 키 제약을 걸어 정합성을 보장합니다. 이 덕분에 **데이터베이스 수준에서 참조 무결성**이 유지되어, 잘못된 외래 키값이 저장되거나 부모 데이터가 없는 자식 데이터가 생성되는 것을 방지합니다. 필요에 따라 JPA 애노테이션 속성이나 설정으로 이러한 외래 키 제약 생성 여부를 조절할 수도 있지만, 기본 설정에서는 논리적인 연관관계에 맞춰 제약을 걸도록 되어 있습니다.

추가적으로, 객체 연관관계 방식은 **양방향 연관관계**를 설정할 수 있다는 점도 특징입니다. 예를 들어 팀 엔티티 입장에서도 자신에게 속한 회원 리스트를 **`@OneToMany`** 컬렉션으로 가질 수 있습니다. 이렇게 양방향 관계를 맺으면 한쪽에서 관계를 추가/삭제할 때 반대편 컬렉션이나 필드에도 일관되게 반영해주어야 하는 등 다소 관리가 필요하지만, 도메인 모델을 풍부하게 표현할 수 있고 JPA가 제공하는 **영속성 전이(cascade)**나 **고아 객체 제거(orphan removal)** 같은 기능을 활용하여 객체 그래프의 생명주기를 한꺼번에 관리할 수 있습니다. 요약하면, 직접 참조 방식은 **객체 지향 모델링의 이점**과 **DB 무결성 보장**을 얻는 방법으로, JPA/Hibernate 환경에서 기본적으로 권장되는 연관관계 맵핑 방식입니다.

## **Long ID 기반 간접 참조 방식**

### 개념

**Long ID 기반 간접 참조 방식**은 엔티티가 다른 엔티티를 직접 객체로 들고 있지 않고, 대신 상대의 식별자 값(ID)만을 저장하는 형태로 연관관계를 관리하는 방법입니다. 여기서 "Long ID"란 주로 데이터베이스의 기본 키로 쓰이는 Long 타입 정수 식별자를 의미합니다. 예를 들어 회원-팀 관계를 간접 참조로 표현한다면, 회원 엔티티 내부에 **`teamId`**라는 Long 필드를 두고 이것을 팀 테이블의 PK 값으로 사용합니다. 객체 자체를 참조하지 않으므로 JPA 입장에서는 이 **`teamId`** 필드는 그저 숫자 컬럼일 뿐이고, 엔티티 간의 공식적인 관계로 인식되지 않습니다. 한 마디로, **데이터베이스 레벨의 외래 키 값**만 엔티티에 저장하고 ORM 차원의 연관관계 매핑은 설정하지 않는 방식입니다.

### 특징

이 접근 방법의 특징은 **느슨한 결합과 단순성**입니다. 각 엔티티 클래스는 자신이 소유한 식별자와 기타 속성들만 가질 뿐, 다른 도메인 객체와의 직접적인 링크를 갖지 않습니다. 덕분에 엔티티 정의가 간결해지고 JPA 연관관계에서 발생하는 복잡성을 피할 수 있습니다.

예를 들어 연관 엔티티를 불러오기 위해 Lazy 로딩 프록시가 작동하거나, 잘못된 연관 설정으로 인한 **N+1 쿼리 문제** 등이 발생할 여지가 없습니다. 회원 엔티티에 team 객체 대신 team의 ID만 있다면, 회원을 조회할 때는 **항상 회원 테이블만 select**하며 팀 정보는 가져오지 않습니다. 팀 정보가 필요할 경우에는 별도로 해당 teamId를 사용하여 팀 레포지토리  등을 통해 **명시적으로 조회**해야 합니다. 이처럼 필요한 순간에 직접 쿼리를 작성하거나 서비스 계층에서 추가 조회를 하는 부담이 있지만, 어떤 데이터가 언제 로드되는지가 개발자에게 보다 **명시적으로 드러난다**는 장점이 있습니다.

Long ID 기반 참조는 주로 **도메인 경계가 뚜렷하게 구분되는 경우**나 **시스템 간 의존도를 낮추고 싶을 때** 유용합니다. 예를 들어 **도메인 주도 설계(DDD)** 관점에서 서로 다른 Aggregation(애그리거트)의 루트는 ID로만 연결하고 직접 객체 참조를 하지 않는 것이 권장되는데, 이때 이러한 ID 참조 방식이 활용됩니다. 하나의 예로, 예약(Appointment) 객체가 의사(Doctor) 객체를 참조해야 한다고 가정하면, 둘 다 각각 독립된 Aggregate Root(애그리거트 루트)이기 때문에 예약에 의사 ID만 저장하고 의사 객체 자체는 두지 않는 식입니다. 이렇게 하면 예약과 의사가 별개로 독립적인 생명주기를 가지며, 한 쪽의 변경이 다른 쪽의 트랜잭션적 일관성에 직접적인 영향을 주지 않습니다. 즉, **각자의 데이터를 각자 관리하고 식별자만 교환**하는 형태로 느슨하게 연결되는 것입니다.

ID 기반 간접 참조 방식에서는 필요하면 데이터베이스 차원에서 외래 키 제약을 수동으로 추가할 수도 있지만, JPA가 이를 자동으로 관리해주지 않으므로 개발자가 DDL에 직접 FK 제약을 명시하거나 마이그레이션 툴을 통해 설정해야 합니다.  만약 DB에 외래 키 제약을 걸지 않는다면, 데이터 무결성은 **어플리케이션 로직에 의존**하게 됩니다.

예를 들어 회원 객체에 잘못된 teamId가 들어가는 것을 막기 위해서는 애플리케이션에서 회원 추가나 수정 시 해당 teamId에 대응되는 팀이 존재하는지 확인하는 검증 로직을 넣어야 합니다. 반대로, 애플리케이션이 이러한 검증을 확실히 수행한다는 전제 하에 DB 레벨 제약을 생략함으로써 얻게 되는 **유연성**도 있습니다. 예를 들어 개발이나 테스트 단계에서 아직 테이블 간 관계가 확정되지 않았거나, 일시적으로 잘못된 참조 데이터가 들어가도 어플리케이션 로직으로 수습할 수 있는 상황이라면 FK 제약 없이 진행할 수 있습니다. 하지만 이런 경우에도 잘못된 참조로 인한 문제(예: 삭제된 팀에 소속된 회원이 남는 문제)를 예방하기 위해 별도의 관리 전략이 필요합니다.

요약하면, Long ID 기반 간접 참조는 **엔티티 간의 결합도를 낮추고 ORM 관련 복잡성을 줄이는 대신, 데이터 정합성 유지와 연관 데이터 조회에 대한 책임을 개발자에게 부과하는 접근법**입니다. 코드 상으로는 각 엔티티가 자기 자신의 정보만 갖고 있기 때문에 단순하고 가볍지만, 연관된 객체를 다루는 작업이 수동으로 이루어져야 하므로 구현상의 수고와 잠재적 오류 가능성을 수반합니다.

## **직접 참조 vs. ID 간접 참조**

두 연관관계 맵핑 방식은 각각 이점과 한계가 분명합니다. 여기서는 **성능**, **유지보수성**, **데이터 정합성**, **유연성** 측면에서 각각의 방식을 비교합니다.

### **성능 측면**

- **쿼리 및 로딩 성능**

  객체 직접 참조를 사용하면 JPA 수준에서 연관 데이터를 가져오기 위한 최적화 기법을 활용할 수 있습니다.

  예를 들어 **지연 로딩(Lazy Loading)**으로 필요할 때만 연관 엔티티를 조회하거나, JPQL에서 **`JOIN FETCH`**를 사용해 한 번의 쿼리로 관련 데이터를 모두 불러올 수 있습니다. 잘 설계된 Fetch 전략을 쓰면 **연관 객체를 한꺼번에 가져와도 성능을 높일 수 있고** N+1 문제도 예방할 수 있습니다. 반면 ID 간접 참조 방식에서는 기본적으로 엔티티 조회 시 연관 데이터는 전혀 가져오지 않으므로 처음에는 매우 가볍습니다. 그러나 정작 연관 정보를 필요로 할 때는 개발자가 별도로 쿼리를 작성해야 하며, 잘못하면 회원 목록을 가져온 뒤 루프를 돌며 개별 teamId로 팀을 재조회하는 **N+1 형태의 부하**를 발생시킬 수 있습니다. 따라서 ID 참조 방식에서는 **필요시 직접 조인 쿼리 작성**이나 한 번의 질의로 여러 객체를 가져오는 Batch 쿼리 등에 대한 고려가 필요합니다. 요약하면, 직접 참조 방식은 ORM이 자동 최적화를 도와주지만 방심하면 숨겨진 비용(lazy N+1 등)이 있고, ID 참조 방식은 초기 쿼리가 단순하고 빠르나 연관 데이터를 사용할 때 명시적 최적화 노력이 든다는 차이가 있습니다.

- **조회 성능 및 인덱스**

  외래 키를 두면 해당 컬럼에 자동으로 인덱스가 생성되거나 적어도 인덱스 생성이 권장됩니다. 이로 인해 **조인 쿼리의 SELECT 성능이 향상**되는 부수 효과가 있습니다. 직접 참조 방식에서는 JPA가 관계를 맺는 순간 DB에 FK와 인덱스를 준비하므로, 회원과 팀을 조인하여 조회하는 쿼리의 성능이 뒷받침됩니다. 반면, ID를 직접 참조하는 방식에서는 해당 필드에 인덱스를 명시적으로 생성하지 않으면, 팀 ID를 기준으로 회원을 조회하거나 조인할 때 쿼리 성능이 저하될 수 있습니다. 따라서 간접 참조 방식을 사용할 경우에는 Flyway와 같은 마이그레이션 도구를 활용해 개발자가 직접 인덱스를 추가해주는 것이 중요합니다. MySQL에서는 외래 키 자체가 조회 성능을 직접 높이기보다는, 외래 키 제약을 만들 때 자동으로 생성되는 인덱스가 성능에 기여합니다. 즉, FK가 없어도 인덱스만 잘 설정되어 있다면 SELECT 성능은 동일하게 유지될 수 있습니다.

- **메모리 사용 및 캐싱**

  JPA는 1차 캐시와 2차 캐시 등을 통해 이미 로드된 객체를 재사용합니다. 직접 참조 관계에서는 한번 불러온 연관 객체를 여러 엔티티가 공유할 수 있으므로 중복 객체를 만들지 않아 메모리 효율성이 좋고, 캐시 혜택을 보기 쉽습니다. 예를 들어 Team A에 속한 회원 100명을 조회하면 Team A 객체는 하나만 생성되어 100명의 Member 엔티티가 같은 객체를 가리키게 할 수 있습니다. ID 참조 방식에서는 회원을 100명 가져오면 teamId 값만 100개 있을 뿐 팀 객체는 로드되지 않았고, 개발자가 별도로 팀들을 조회하면 그때서야 팀 객체들이 생성됩니다. 이 과정에서 적절히 캐싱을 구현하지 않으면 동일 팀을 여러 번 읽어 메모리를 낭비하거나 성능을 떨어뜨릴 수 있습니다. 반면, 필요할 때까지 아예 객체 생성을 안 하므로 그만큼 초기 메모리 소비는 낮다는 장점도 있습니다. 또한 ID 참조 방식에서도 JPA 2차 캐시 등을 이용해 **`findById`**를 할 때 캐시된 객체를 얻어올 수 있으므로, 개발자가 수동으로 조회하는 경우라도 캐시 활용은 가능합니다.


정리하면, **성능 면에서는 직접 참조 방식이 ORM의 자동 최적화와 DB 인덱스 혜택으로 효율적인 반면, 부적절한 사용 시 숨은 비용이 발생할 수 있고**, ID 참조 방식은 초기 쿼리가 가볍고 의도치 않은 N+1은 없으나 **연관 데이터를 사용할 때 개발자의 명시적 최적화가 필요**합니다.

### **유지보수성과 개발 생산성 측면**

- **코드의 이해도와 복잡도**

  객체 직접 참조는 도메인 모델 간의 관계를 코드 상에 명시적으로 드러내므로, 해당 코드를 읽는 개발자가 비즈니스 연관관계를 쉽게 파악할 수 있습니다. 엔티티 안에 Team 필드가 있다면 "회원은 팀을 가진다"는 도메인 지식을 바로 알 수 있는 반면, teamId만 있다면 주석이나 문서를 보지 않는 한 이 값이 무엇을 의미하는지, 어떤 테이블과 연관되는지 유추해야 합니다. 또한 JPA 연관관계를 적극 활용하면 도메인 로직을 구현할 때 **직관적인 객체 그래프 탐색**으로 비즈니스 로직을 표현할 수 있어 생산성이 높아집니다.

  예를 들어 **`member.getTeam().getManager()`**처럼 연쇄적으로 메서드를 호출하며 비즈니스 의미를 드러낼 수 있습니다. 반대로 ID 참조 방식에서는 **`member.getTeamId()`**로 값만 얻은 뒤 별도의 서비스나 리포지토리를 호출해서 팀을 가져와야 하므로, **비즈니스 로직 내에 데이터 접근 코드가 섞이기 쉽고** 코드량이 늘어납니다. 이는 계층 간 책임 분리가 잘 되어 있는 경우에도 추가적인 노력이 필요함을 의미합니다. 결과적으로 직접 참조 방식이 **도메인 모델을 풍부하게** 해주고 코드 가독성을 높여주는 측면이 있습니다.

- **ORM 활용에 따른 복잡성**

  직접 참조를 사용하면 JPA의 다양한 기능(캐스케이드 저장, 고아 제거, Lazy/Eager 로딩 설정 등)을 함께 관리해야 합니다. 이는 곧 개발자가 **ORM의 동작을 잘 이해하고 있어야** 함을 뜻합니다. 예를 들어 양방향 관계일 때 연관관계의 주인(owner) 개념, **`mappedBy`** 설정 등을 숙지해야 하며, 한쪽 엔티티를 저장하거나 삭제할 때 다른 쪽에 미치는 영향을 예측해야 합니다. 이러한 학습 곡선과 설정상의 복잡함 때문에, **연관관계를 잘못 맵핑하여 발생하는 버그나 성능 문제**도 흔히 나타납니다. 반면 ID 참조 방식은 엔티티가 서로 몰랐다는 듯이 독립적이므로 JPA 관계 설정으로 인한 오류는 애초에 없습니다. 엔티티 간 결합이 없으니 **JPA를 비교적 단순한 CRUD 도구처럼** 활용할 수 있고, 복잡한 맵핑보다는 SQL에 가까운 사고 방식으로 개발을 진행할 수 있습니다. 작은 팀이거나 JPA 숙련도가 높지 않은 상태에서 빠르게 개발해야 할 때 이러한 접근을 택하면 **학습 비용이나 실수 위험을 낮출 수도 있습니다**.

- **도메인 모델 변화에 대한 대응**

  시간이 지나 도메인 간 관계가 변경되거나 새로운 요구사항이 추가되는 경우, 직접 참조 방식은 관련된 엔티티 매핑을 수정하고 데이터 마이그레이션을 동반하는 구조적 변경이 필요합니다. 예를 들어, 회원이 반드시 팀에 소속된다는 가정 하에 `@ManyToOne(optional = false)`로 설정해 두었다면, 팀 없이도 가입이 가능한 구조로 바꾸려면 애노테이션 옵션을 수정하고 스키마까지 변경해야 합니다.

  반면, 간접 참조 방식은 구조적으로 유연합니다. 연관 관계를 강하게 고정하지 않기 때문에 도메인 간 연결 방식이 바뀌더라도 ID 처리 로직만 조정하면 비교적 간단하게 대응할 수 있습니다. 물론 직접 참조는 정적 타입 기반이라 컴파일 시점에 타입 오류를 잡아낼 수 있고, 관련 코드도 명시적으로 연결되어 있어 변경 범위를 추적하기 쉬운 장점이 있습니다. 그러나 도메인 간 관계가 자주 변경되거나, 처음부터 관계가 명확하지 않은 경우라면 간접 참조가 변경 대응 관점에서 관리 부담을 덜어줄 수 있습니다.


요약하자면, **유지보수성 측면에서는** 객체 직접 참조 방식이 초기 구현과 코드 이해 면에서 개발자에게 친화적이고 풍부한 모델링을 제공하지만, 이를 제대로 다루기 위한 JPA 지식이 필수이고 설정이 복잡해질 수 있습니다. Long ID 간접 참조는 단순한 구조로 실수를 줄이고 엔티티 자체는 깔끔하게 유지할 수 있으나, **관계에 대한 제약과 로직이 눈에 보이지 않는 곳에서 처리**되어야 하기 때문에 체계적인 관리가 동반되어야합니다.

### **유연성과 확장성 측면**

- **아키텍처 변화와 모듈화**

  ID 기반 간접 참조의 가장 큰 장점 중 하나는 시스템을 **모듈화하거나 마이크로서비스로 분리**할 때 드러납니다. 엔티티들이 서로 직접 연결되어 있지 않고 식별자만 주고받기 때문에, 만약 회원과 팀 도메인을 나누어 별개의 서비스로 이전한다 해도 비교적 수월하게 분리가 가능합니다. 한 서비스에서는 teamId만 가지고 있고, 실제 팀에 대한 조회는 다른 서비스의 API를 호출하는 식으로 대체할 수 있기 때문입니다. 반면 직접 객체 참조로 엮여 있는 경우, 모듈을 분리하려면 그 사이의 연관관계를 끊어내야 하고 관련된 도메인 로직도 함께 수정해야 합니다. 즉, **ID 참조 방식은 초기부터 경계를 명확히** 해놓는 효과가 있어 **미래의 구조 변화에 대비한 유연성**이 높습니다. 이러한 이유로, 처음에는 하나의 모놀리식 애플리케이션이지만 장차 서비스 분리를 염두에 두고 있다면 연관 도메인 간에는 ID 참조를 사용하자는 전략을 취하기도 합니다.

- **스키마 및 기능 확장의 용이성**

  새로운 요구사항이 생겨서 데이터 모델을 변경해야 할 때, ID 참조 방식은 제약이 적어 변형이 비교적 쉽습니다. 예를 들어 회원이 팀 대신 부서(Department)를 가리키도록 변경된다면, 직접 참조를 써둔 경우 **`Member.team`** 필드를 **`Member.department`**로 바꾸고 모든 관련 JPQL, 서비스 로직, FK 제약 등을 다 수정해야 합니다. ID만 쓰고 있었다면, **`teamId`** 필드 이름을 **`departmentId`**로 바꾸는 식의 단순 변경으로 끝날 수도 있습니다(물론 그 이상의 도메인 로직 수정은 필요하겠지만, 적어도 JPA 맵핑 차원에서는 단순합니다). 또한 데이터베이스 차원에서 외래 키 제약이 없으면 **테이블 변경이나 데이터 이동 작업을 할 때 제약을 일시 해제하거나 드롭하는 절차 없이** 진행할 수도 있습니다. 이러한 점들은 개발 및 운영 상의 유연성을 높여줍니다.


결론적으로, **유연성과 확장성 측면에서는** ID 간접 참조 방식이 **시스템 분리와 진화에 더 유연하게 대처**할 수 있고, 스키마나 기능 변경에 덜 제약을 받는 장점이 있습니다. 반면 객체 직접 참조 방식은 **엄격한 규칙 준수와 일관성을 대가로 약간의 경직성을 갖지만**, 대부분의 일반적인 애플리케이션에서는 이 "경직성"이 오히려 올바른 설계를 유지하는 가드레일 역할을 한다는 점도 고려해야 합니다.

## **도메인 생명주기와 연관관계 설계**

지금까지 객체 참조와 간접참조의 개념과 특징을 비교해보았습니다. 이 둘이 서로를 대체하는 방식처럼 여겨지지만, 실제로는 **배타적인 선택지가 아니라 도메인의 성격에 따라 공존할 수 있는 설계 전략**입니다. 프로젝트 내 여러 도메인 각각이 가진 특성에 맞춰, 직접 참조와 간접 참조를 혼합하여 사용하는 것이 보다 현실적인 접근입니다.

핵심은 **관계를 맺는 엔티티들이 생명주기를 얼마나 공유하는지**, 다시 말해 **함께 생성되고 함께 사라지는 운명 공동체 관계인지**, 아니면 **서로 독립된 생애를 가진 별개의 주체인지**를 파악하는 데 있습니다.

### **함께 생성되고 함께 소멸되는 관계: 강한 생명주기 공유**

두 객체가 생성과 소멸을 같이하며, 한쪽이 없으면 다른 쪽도 의미를 잃는 경우라면 **객체 직접 참조**를 사용하는 것이 자연스럽습니다. 예를 들어 `Order`와 `OrderItem`의 관계를 생각해볼 수 있습니다. 주문 항목은 반드시 어떤 주문에 속해야 하며, 주문이 삭제되면 관련 항목도 함께 사라져야 합니다. 이처럼 명확한 소유 관계가 있는 경우에는 JPA의 cascade 기능과 `orphanRemoval` 옵션을 활용해 하나의 단위로 관리하는 것이 적합합니다.

DB 측에서도 외래 키에 `ON DELETE CASCADE`를 설정하여 삭제 전파가 일어나게 할 수 있습니다. 반면 이처럼 생명주기가 강하게 연결된 관계에서 ID만 따로 보관하도록 설계하면, 객체 간 일관성을 애플리케이션 단에서 일일이 관리해야 하므로 오히려 관리 비용이 증가할 수 있습니다.

### **독립된 생명주기를 가진 객체들: 느슨한 연결을 고려**

반대로 두 객체가 독립적으로 존재하며, 삭제나 생성이 반드시 함께 일어나지 않아도 되는 경우라면 **ID 참조 방식이 더 유연한 설계가 될 수 있습니다**. 예를 들어 회원(Member)과 팀(Team)은 명확히 분리된 생명주기를 가집니다. 팀이 삭제되더라도 회원이 반드시 같이 삭제되어야 할 이유는 없습니다. 오히려 회원이 다른 팀으로 옮겨가거나 팀 없이 존재하는 상황도 가능합니다.

이런 경우에도 객체 직접 참조 + FK 제약으로 관계를 구성할 수는 있지만, 팀 삭제 시 제약 조건 위반이 발생하고 회원 테이블의 데이터를 일괄 수정해야 할 수 있습니다. 특히 팀 통폐합이나 삭제가 잦은 도메인에서는 이런 강한 연결이 반복적인 유지보수 부담으로 이어질 수 있습니다. 대신, **간접 참조로 구성하고 삭제 시 `teamId`만 null 처리하는 방식**을 선택하면 훨씬 유연한 구조가 될 수 있습니다.

물론 이 과정에서 데이터 정합성을 유지하기 위해선 팀 삭제 시 관련 회원 데이터를 적절히 업데이트하는 로직이 필요합니다. 하지만 설계 상 두 객체의 생명주기가 다르다면, 이러한 느슨한 연결이 오히려 현실적인 대응일 수 있습니다.

### **애그리거트 간의 참조는 ID로, 내부에서는 직접 참조로**

도메인 주도 설계(DDD) 관점에서 보면, **애그리거트(Aggregate) 내부는 직접 참조**, 애그리거트 간에는 **ID를 통해 느슨하게 참조하는 방식**이 권장됩니다. 예를 들어 `Order` 애그리거트 내부에서 `OrderItem`은 객체 참조로 직접 다루되, `Order`가 다른 애그리거트인 `User`를 참조해야 한다면 해당 유저의 ID만 저장하는 방식입니다.

이렇게 하면 각 애그리거트가 독립적으로 트랜잭션을 갖고 자체적인 비즈니스 규칙을 유지할 수 있으며, 도메인 경계도 명확히 분리됩니다. 한 트랜잭션 안에서 둘 이상의 애그리거트를 동시에 수정해야 한다면, 그것은 경계 설정이 잘못된 신호일 수 있습니다. 이러한 경계 의식을 설계에 녹여두면 확장성과 유지보수성 모두에 긍정적인 영향을 줄 수 있습니다.

### **관계의 강도에 따라 전략은 달라질 수 있습니다**

- 생명주기를 **강하게 공유**한다면 → **직접 참조 + cascade + FK 제약**
- 생명주기가 **독립적**이라면 → **ID 참조 + 느슨한 연결**
- 같은 애그리거트 안에서는 → **객체 참조로 깊게 연결**
- 애그리거트 간에는 → **ID로만 연결하여 경계 분리 유지**

설계는 결국 **도메인의 성격과 변화 가능성을 중심으로 판단**해야 하며, 
같은 시스템 안에서도 도메인마다 전략이 달라질 수 있습니다. 
중요한 것은 양쪽 방식의 특성과 한계를 충분히 이해하고, 도메인별로 적절한 판단을 내리는 것입니다.
