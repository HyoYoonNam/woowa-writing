# 소규모 서비스에서 효율적인 검색 기능 구축하기


## 1. 검색 기능 도입의 배경과 초기 고민
새롭게 서비스를 정식 출시하면서 검색 기능을 제대로 구현해야 하는 과제가 생겼습니다. 
사용자들이 원하는 정보를 쉽게 찾을 수 있도록 검색을 강화하고자 했고, 최종 목표는 뉴스레터 플랫폼 캐릿(CAREET)처럼 검색어가 제목과 본문에 몇 번 언급되었는지까지 보여주는 것이었습니다   


캐릿에서는 특정 키워드를 검색하면 해당 키워드가 제목에 포함된 콘텐츠를 최상단에 노출하고, 본문에서 해당 키워드가 가장 많이 언급된 콘텐츠를 그 다음 순위로 노출하는 방식을 취합니다


우리 서비스도 이러한 기능을 참고하여, 키워드가 많이 등장하는 콘텐츠일수록 사용자가 찾는 주제와 더 밀접할 것이라는 판단을 했습니다. 현 상태에서 우리의 검색 기능은 제목만 대상으로 검색이 이루어지고 있었습니다. 예를 들어 어떤 키워드로 검색하면 제목에 그 단어가 포함된 아티클은 잘 나오지만, 본문 내용에만 있는 경우는 검색되지 않는 한계가 있었죠. 이는 서비스 구조상 뉴스레터 아티클의 본문이 매우 길고 (하나의 아티클에 여러 주제와 많은 양의 글이 포함됨), 단순한 방법으로 본문까지 모두 검색할 경우 성능 문제가 우려되었기 때문입니다. 우선 현재 데이터를 분석해 보았습니다. 뉴스레터 아티클은 HTML 형식으로 메일이 오며, 이를 파싱하여 DB에 그대로 저장하고 있습니다. 하나의 아티클은 HTML 태그 포함 시 약 8만~16만 자 정도의 길이이고, 태그를 제거한 순수 텍스트만 약 5천~1만5천 자 수준입니다. 대략 10배 정도의 차이가 나기 때문에, 검색 시에는 불필요한 HTML 태그를 제외한 텍스트만 사용하기로 했습니다. 따라서 DB에 원본 HTML과 별도로 검색용 순수 텍스트를 함께 저장해두는 방식으로 데이터를 관리했습니다. 그래야 검색시 불필요한 태그 처리 없이 텍스트만 빠르게 검색할 수 있기 때문입니다. 그렇다고 해도 수천 자에 달하는 본문 텍스트를 대상으로 검색을 하면 부하가 만만치 않습니다. 단순히 SQL의 LIKE 연산자로 %키워드%를 검색한다면 인덱스를 탈 수 없고 테이블 전체를 훑는 식이어서, 가장 데이터가 많은 이 아티클 테이블에서는 성능 저하가 불 보듯 뻔했습니다. 검색 기능을 도입하기 전에, 서비스 특성과 데이터를 철저히 파악하고 (말 그대로 지피지기 백전불태의 마음으로) 어떤 검색 구현 방식을 선택할지 고민하기 시작했습니다.

## 2. 검색 구현 방법 선택: MySQL 풀텍스트 인덱스
 
### 1) 전용 검색 엔진 vs. MySQL – 무엇을 선택할까?
   검색 기능을 강화하기 위해 선택지는 크게 두 가지였습니다. 
   하나는 Elasticsearch와 같은 전용 검색 엔진을 도입하는 것이고, 다른 하나는 현재 사용 중인 MySQL 기능을 활용하는 것이었습니다. 먼저 전용 검색 엔진을 검토해보았는데, 우리처럼 소규모 서비스에게는 상당한 부담이 있었습니다. Elasticsearch의 경우 Java 기반으로 동작하며 메모리와 CPU를 많이 요구하는데, 공식 문서에도 운영 환경에서는 최소 8GB RAM을 권장하고 (테스트 용도로도 4GB 정도는 필요합니다), 스왑 메모리 사용도 지양하라고 합니다. 현재 우리의 서버는 AWS EC2의 T4g.micro 인스턴스로 RAM이 1GB에 불과합니다. 아직 수익화도 되지 않은 상황에서 검색 서버를 위해 인스턴스 사양을 크게 높이거나 별도 클러스터를 운영하는 것은 현실적으로 어려웠습니다. 게다가 Elasticsearch를 새로 도입하면 러닝 커브도 있고 운영 부담도 커지므로, 서비스 초반에는 기존 데이터베이스로 최대한 해결하는 쪽을 택하기로 했습니다. 남은 방안은 MySQL에서 검색을 구현하는 것입니다. 가장 간단한 방법인 LIKE 연산자는 앞서 언급했듯 와일드카드(%키워드%) 사용 시 인덱스를 사용할 수 없어 대용량 텍스트 검색에는 부적합했습니다. 다행히 MySQL에는 전문 검색(Full-Text Search) 기능이 있습니다. MySQL의 **풀텍스트 인덱스(fulltext index)**는 텍스트 검색을 빠르게 해주는 전문 인덱스로서, 일반적인 단어 검색에 특화된 기능입니다. 우리는 결국 MySQL 풀텍스트 인덱스를 활용하는 방향으로 가닥을 잡았습니다.
   
### 2) MySQL 풀텍스트 인덱스와 N-gram 파서 이해
   MySQL 풀텍스트 검색을 사용하려면 풀텍스트 인덱스를 먼저 생성해야 합니다. 
   인덱스 없이 MATCH ... AGAINST 쿼리를 실행하면 에러가 발생하므로, 검색 대상 컬럼에 대해 미리 인덱스를 만들어두어야 합니다. 풀텍스트 인덱스는 일종의 **역색인(inverted index)**으로, 문서 내 단어들을 빠르게 찾아낼 수 있는 자료 구조입니다. 문제는 한국어였습니다. MySQL의 기본 풀텍스트 검색은 영어 등 띄어쓰기로 단어 구분이 명확한 언어에 최적화되어 있습니다. 예컨대 "My cat is cute"라는 문장이 있으면 공백을 기준으로 [My], [cat], [is], [cute] 같은 단어 토큰으로 쪼개서 인덱싱합니다. 하지만 한국어는 띄어쓰기만으로 단어 경계가 분명하지 않고, 조사나 접사가 붙어 단어 형태가 변합니다. 이런 언어에는 MySQL 8.0 이상에서 제공하는 N-gram 파서를 적용하는 것이 효과적입니다. N-gram 파서란 텍스트를 인덱싱할 때 일정 길이로 잘라서 토큰을 만드는 방식입니다. 예를 들어 2-gram(bi-gram)으로 "데이터 분석은 재미있다"라는 문장을 파싱하면 다음과 같은 토큰들이 만들어집니다:
   "데이터", "이터분", "터분석", "분석은", "석은재", "은재미", "재미있", "미있다"
   이처럼 연속된 2글자씩 겹치도록 잘라내므로, 단어 내부에 있는 조각들도 인덱스로 기록됩니다. 이렇게 하면 설령 정확히 일치하는 단어가 아니더라도 일부 문자열 매칭으로 검색이 가능해집니다. 예를 들어 본문에 "데이터분석"이라는 단어가 있더라도, 사용자가 "데이터"만 검색해도 "데이터분석"의 일부가 일치하는 것으로 검색 결과에 나올 수 있는 식입니다. 한글, 일본어, 중국어 등 공백으로 명확히 단어를 구분하기 어려운 언어에서는 이러한 N-gram 방식이 검색 품질을 높여줍니다.

> 📌 풀텍스트/N-gram 설정 체크리스트
> - `innodb_ft_min_token_size=2` (2자 이상 토큰화)
> - `ngram_token_size=2` (한글 2-gram 권장)
> - `innodb_ft_enable_stopword=OFF` 또는 커스텀 스톱워드 테이블
> - 위 설정 변경 후 **FULLTEXT 인덱스 재생성 필수**
>
> 예시:
> ```sql
> -- (RDS: 파라미터 그룹으로 설정 권장) 이후 인덱스 재생성
> ALTER TABLE article_recent DROP INDEX ft_recent_title_body;
> ALTER TABLE article_recent
>   ADD FULLTEXT KEY ft_recent_title_body (title, body_text) WITH PARSER ngram;
> ```

### 3) N-gram의 비용: 인덱스 크기와 성능 상의 고려
   N-gram 파서는 강력하지만 대가를 치러야 합니다. 바로 인덱스 크기가 기하급수적으로 증가할 수 있다는 점입니다. N-gram으로 텍스트를 잘게 쪼개면 그만큼 조각이 많이 생겨서, 인덱스 데이터의 용량이 원문 대비 200~300% 이상 커질 수 있습니다. 경우에 따라서는 수십 배에서 수백 배까지 커지기도 하는데, 문장이 길수록 N-gram 조각들이 폭발적으로 늘어나기 때문입니다. 예컨대 태그를 제외한 본문이 약 1만 자라면, 2-gram 토큰은 1만 개보다 훨씬 많이 생성되어 인덱스가 원문 크기의 몇 배에 달할 수 있습니다. 인덱스가 커지면 메모리와 저장소에 부담이 갑니다. MySQL(InnoDB)의 인덱스는 기본적으로 디스크에 저장되지만, 자주 사용되는 인덱스 페이지들은 메모리(RAM)의 버퍼 풀에 올라와 캐시됩니다. 메모리에 더 많이 올라올수록 실제 쿼리 속도는 비약적으로 빨라집니다. 반대로 인덱스가 너무 커서 메모리에 다 올려두지 못하면, 검색할 때마다 디스크에서 필요한 부분을 읽어와야 하므로 디스크 I/O로 인한 지연이 발생합니다. 특히 이런 랜덤 I/O는 검색 성능을 급격히 떨어뜨립니다. 즉, 인덱스는 공짜가 아니며 메모리는 한정된 자원입니다. 모든 텍스트 컬럼에 죄다 인덱스를 걸거나, N-gram처럼 인덱스 용량을 부풀리는 방법을 남발하면 오히려 성능이 악화될 수 있습니다. 필요한 최소한의 인덱스만 두고, 특히 대용량의 특수 인덱스(N-gram 등)는 신중하게 사용해야 합니다. 우리 서비스의 경우 사용 가능한 메모리가 매우 제한적이라, N-gram 인덱스를 도입하면 성능 이득보다 손해가 클 위험을 인지했습니다. 이를 해결하기 위해 어느 부분에 N-gram을 적용할지 범위를 최적화하기로 했습니다.

| 리소스 항목 | 현재 값/한계 |
|---|---|
| DB 인스턴스 | T4g.micro (RAM 1GB) |
| InnoDB Buffer Pool | 기본 128MB (RDS 최대 약 384MB) |
| N-gram 인덱스 범위 | 최근 5일(예상 ~200MB 내외) |
| 위험 신호 | Buffer Pool Hit Ratio 하락, 랜덤 I/O 증가, P95 지연 상승 | 

## 3. N-gram과 검색 모드 최적화 전략
   
### 1) N-gram 적용 범위 제한: 최근 데이터에 집중
   서비스 도메인을 살펴보면 뉴스레터 콘텐츠는 시의성이 강합니다. 새로운 뉴스는 매일 쏟아지고, 일주일이나 한 달만 지나도 정보 가치가 떨어지는 경우가 많습니다. 반면 사용자들이 검색을 통해 얻고자 하는 정보는 최근 이슈나 최신 자료일 가능성이 높죠. 이러한 특성을 고려하여, N-gram 인덱스는 최신 글에만 제한적으로 적용하기로 결정했습니다. 모든 아티클에 N-gram 인덱스를 거는 것은 리소스 낭비이므로, 일정 기간 내의 데이터만 부분 검색이 가능하도록 하고, 그 기간이 지난 데이터는 N-gram 없이도 충분히 서비스 운영이 가능하다고 판단했습니다. 구체적으로 어떤 기간을 최신 데이터로 볼 것인가가 문제였습니다. 이를 결정하기 위해 현재 사용 중인 환경의 한계를 따져보았습니다. 우리 서비스는 AWS RDS의 MySQL을 사용하고 있으며, 인스턴스 스펙은 T4g.micro (RAM 1GB)입니다. InnoDB 버퍼 풀은 기본값인 128MB로 설정되어 있었는데, RDS에서 허용하는 최대값이 약 384MB 정도였습니다 (RDS가 인스턴스 메모리 대비 안전선을 두고 제한). 즉, 아무리 설정을 조정해도 DB 캐시로 384MB 이상은 사용할 수 없는 상황이었습니다. 이제 N-gram 인덱스 크기를 대략 계산해보았습니다. 가정을 단순화하기 위해, 활성 사용자 200명이 매일 각 5개의 뉴스레터를 받는다고 해봅시다. 뉴스레터 하나당 순수 텍스트 길이 1만 자로 잡겠습니다. N-gram을 적용하면 인덱스가 원문 대비 2~3배 이상 불어날 수 있으므로, 하루치 뉴스레터에 약 10만 자 상당의 인덱스가 생길 수 있습니다 
   최근 7일치 데이터에 N-gram 인덱스를 걸면 약 280MB 정도의 용량이 필요할 것으로 추산되었습니다.
   1개월치에 적용하면 약 1.2GB까지 불어날 수 있습니다.
   1년치 모든 데이터에 N-gram 인덱스를 적용하면 무려 14~15GB 규모로 커질 것으로 예상되었습니다.
   
## 4. N-gram 적용과 검색 모드 조합 전략

### 1) 최신 데이터에만 N-gram 인덱스 적용하기
   이제 N-gram 인덱스를 어디까지 적용할지 범위를 정해야 했습니다. 우리는 최신 뉴스레터 데이터에만 N-gram을 사용하고, 그 기간이 지난 데이터에는 기본 풀텍스트 검색을 사용하기로 했습니다. 뉴스레터 특성상 새로운 정보는 최신 호수에 몰려 있고, 지난 호는
   빈도가 낮아집니다. 최신 이슈에 대한 검색 정확도를 높이는 편이 효율적이라 판단한 것입니다. 문제는 '최신'의 기준을 어떻게 잡느냐였습니다. 우리 서비스가 사용하는 AWS RDS(MySQL)
   니다. 현재 인스턴스는 **RAM 1GB (T4g.micro)**로 버퍼 풀 캐시가 기본 128MB(최대 약 384MB) 정도였습니다. 이 제한 내에서 N-gram 인덱스가 감당될 만한 기간을 산정해 보았습니다. 대략적인 계산으로, 7일치 아티클에 N-gram 인덱스를 적용하면 약 280MB 정도의 인덱스 용량이 필요했습니다. 1개월(30일) 분량에 대해서는 약 1.2GB, 1년치 전체에 적용하면 무려 14~15GB 규모로 불어날 것으로 추산되었습니다. 메모리 캐시 한계를 고려하면 7일치도 다소 벅찰 수 있고, 1개월 이상은 거의 불가능한 수준입니다. 따라서 우리는 보수적으로 최신 5일 이내의 데이터만 N-gram 인덱스를 적용하기로 했습니다. 5일 분량이면 대략 200MB 안팎으로 예상되어, 메모리 384MB 선에서 비교적 안전하다고 판단했습니다.

### 2) 과거 데이터 검색: 불리언 모드로 부분 검색 지원
   N-gram을 최근 며칠에만 적용한다면, 5일 이후의 과거 데이터에 대해서는 기본 풀텍스트 검색을 사용해야 합니다. 그런데 기본 **자연어 모드(NATURAL LANGUAGE MODE)**로는 한글 단어의 일부만 일치하는 검색이 어렵습니다. MySQL 풀텍스트 검색에는 세 가지 모드가 있는데:
   NATURAL LANGUAGE MODE: 기본 모드로, 검색어를 자동으로 분석하고 관련도 순으로 결과를 정렬합니다.
   BOOLEAN MODE: AND, OR, NOT 등의 불리언 연산자와 와일드카드(*) 등을 사용해 수동으로 제어하는 모드입니다.
   WITH QUERY EXPANSION: 한 단계 더 나아가 검색어와 연관된 단어까지 확장해서 검색하는 모드인데, 영어에 특화되어 있어 한글에는 적합하지 않습니다.
   우리는 이 중에서 BOOLEAN 모드에 주목했습니다. Boolean 모드는 검색어에 와일드카드(*)를 접미사로 붙여서 접두사 일치 검색을 할 수 있다는 장점이 있습니다. 예를 들어 본문에 "경제학개론"이라는 단어가 있을 때, 기본 자연어 모드로 "경제학"을 검색하면 해당 문서를 찾지 못하지만, 불리언 모드에서 "경제학*"라고 검색하면 "경제학개론"처럼 '경제학'으로 시작하는 단어도 매치시킬 수 있습니다. 마찬가지로 "배임죄"라는 단어가 있을 경우 자연어 모드로 "배임"을 찾으면 결과가 나오지 않지만, 불리언 모드로 "배임*"이라고 하면 "배임죄"를 포함한 문서를 찾을 수 있습니다. 이처럼 불리언 모드는 한글 검색에 유리하고, 사용자들이 완색하는 현실적인 사용 패턴을 지원해줍니다. 물론 이 방식에도 한계는 있습니다. 와일드카드를 접두사에만 사용할 수 있기 때문에, 단어의 끝부분이나 중간 일부로는 검색이 되지 않습니다. 예컨대 본문에 "인공지능"이 있을 때 사용자가 "공지능"으로 검색하면 (어미가 일치하는 경우) 마찬가지로 검색되지 않는 문제가 남습니다. 하지만 접미사 매칭까지 지원하려면 범위 검색이나 N-gram 같은 방법이 다시 필요해져서 복잡도가 급증합니다. 우리는 현 단계에서는 접두사 일치만 지원해도 검색 편의성이 크게 개선된다고 보고, 이 정도 제약은 감수하기로 했습니다.

## 5. 최종 구현: 최신 5일은 N-gram, 이후는 불리언 모드
   고민 끝에 완성된 검색 기능 구현 방안은 다음과 같습니다.
   최근 5일 이내의 아티클: 제목 + 본문 컬럼에 N-gram 파서 기반의 풀텍스트 인덱스를 적용했습니다. 이를 통해 해당 기간의 콘텐츠에 대해서는 키워드의 부분 일치 검색도 빠르게 지원됩니다. 사용자가 입력한 검색어가 본문에 일부만 포함되어 있어도 N-gram 인덱스를 통해 결과를 찾아낼 수 있습니다.
   발행 후 5일이 지난 아티클: 제목 컬럼에만 풀텍스트 인덱스를 두고, 불리언 모드로 검색을 처리합니다. 오래된 콘텐츠에 대해서는 제목에 키워드가 정확히 (또는 접두어 형태로) 포함된 경우에 한해 검색되도록 했습니다. 본문 전체에 대해 인덱스를 유지하지 않으므로 리소스를 절약할 수 있고, 대신 제목이라는 짧은 텍스트 내에서의 정확한 검색으로 성능을 확보했습니다.
   이 전략으로, 최신 뉴스레터의 풍부한 정보를 부분 검색으로 제공하면서도 오래된 방대한 데이터에 대한 검색 부하는 효과적으로 통제할 수 있었습니다. 5일 이후의 콘텐츠는 제목에 키워드가 없다면 검색에 노출되지 않는 제한이 있지만, 앞서 논의한 대로 최신 정보 중심의 서비스 특성상 큰 문제는 없다고 보았습니다. 결과적으로 사용자는 관심 키워드가 최근에 많이 언급된 콘텐츠를 쉽고 빠르게 찾을 수 있고, 서비스는 제한된 자원 내에서 원활한 검색 성능을 유지할 수 있게 되었습니다.

## 6. 향후 개선 방향: 형태소 분석 도입 검토
   검색 기능을 고도화하는 중장기 전략으로는 한국어 형태소 분석기의 도입을 고려하고 있습니다. 현재의 N-gram 방식은 부분 검색을 가능하게 해주지만 그 대가로 인덱스 크기가 크게 늘어난다는 문제점이 있었습니다. 이를 해결하기 위해 형태소 단위로 텍스트를 분할해 인덱싱하면, 불필요한 조각을 줄이고도 충분한 검색 품질을 얻을 수 있습니다. 예를 들어 형태소 분석을 도입하면 "경제학개론"이라는 단어는 "경제학"+"개론"처럼 의미 단위로 쪼개어 인덱싱됩니다. 이렇게 하면 "경제학"으로 검색해도 해당 문서를 찾을 수 있고, "개론"으로 검색해도 연결될 수 있습니다. 불용어(조사 등)를 걸러내고 실질적인 단어만 인덱싱하기 때문에 인덱스 용량을 효율적으로 줄일 수 있고 검색 정확도도 높아집니다. 이미 일부 오픈소스 형태소 분석 플러그인(예: MeCab-ko 등)을 MySQL과 연동하는 사례도 있습니다. 다만 AWS RDS와 같은 관리형 DB 환경에서는 사용자가 직접 플러그인을 설치할 수 없다는 제약이 있습니다. 따라서 이를 활용하려면 별도의 MySQL 서버를 구성하거나, 아예 Elasticsearch 등의 전문 검색엔진으로 전환해야 합니다. 현 시점에서는 우리 서비스 규모에 비해 과한 작업이기 때문에 도입하지 않았지만, 추후 트래픽이 크게 증가하고 검색 수요가 늘어난다면 형태소 분석 기반의 검색으로 업그레이드하는 것도 고려하고 있습니다.
## 7. 맺으며: 작은 서비스에서 찾은 실용 해법
   이번에 구축한 검색 기능은 소규모 서비스의 현실적인 제약 속에서 최선의 균형점을 찾은 사례라고 할 수 있습니다. MySQL 풀텍스트 인덱스와 한글 N-gram 파서를 적재적소에 활용하고, 불리언 검색 모드로 부족한 부분을 보완함으로써 별도의 검색 서버 없이도 웬만한 기능을 구현할 수 있었습니다. 핵심은 우리 서비스의 사용 패턴과 데이터 특성을 정확히 파악하고, 거기에 맞춰 리소스 활용을 최적화하는 전략을 세운 것입니다. 신입 개발자 분들께 드리고 싶은 조언은, 항상 최신 기술이나 스케일 큰 솔루션이 정답은 아니라는 것입니다. 이번 사례처럼 현재 사용 중인 도구의 숨은 기능들을 찾아보고 창의적으로 적용하면, 제한된 환경에서도 충분히 유용한 결과를 얻을 수 있습니다. 작은 서비스일수록 경량화된 설계와 운영 비용 절감이 중요합니다. 우리의 검색 기능 도입기가 여러분께 비슷한 도전을 마주했을 때 실용적인 해결책을 모색하는 데에 작은 인사이트가 되길 바랍니다.
